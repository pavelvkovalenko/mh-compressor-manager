# Процедура «Проведи полный аудит» — mh-compressor-manager

## Назначение

Данный файл фиксирует текущую логику действий AI-ассистента при получении команды
**«проведи полный аудит»**.

## Текущая логика выполнения

### Шаг 1: Параллельный запуск агентов для чтения файлов

AI-ассистент запускает **два параллельных `agent`-процесса** (subagent) для чтения исходного кода.

**Агент 1:**
- Читает файлы: `compressor.cpp`, `compressor.h`, `main.cpp`
- Задача: найти ошибки в логике сжатия, streaming, one-shot, обработке ошибок, ресурсах

**Агент 2:**
- Читает файлы: `monitor.cpp`, `monitor.h`, `threadpool.h`, `config.cpp`, `config.h`, `logger.cpp`, `logger.h`
- Задача: найти ошибки в логике мониторинга, пула потоков, конфигурации, логирования

**Агент 3:**
- Читает файлы: `async_io.cpp`, `async_io.h`, `memory_pool.h`, `security.cpp`, `security.h`, `performance_optimizer.cpp`, `performance_optimizer.h`, `pipeline.cpp`, `pipeline.h`, `numa_utils.cpp`, `numa_utils.h`, `cache_info.cpp`, `cache_info.h`
- Задача: найти ошибки в async I/O, пуле памяти, безопасности, pipeline, NUMA, cache_info

> ⚠️ **Проблема текущей логики:** Агенты работают параллельно и независимо.
> Каждый агент читает только часть файлов и может пропустить ошибки, которые видны
> только при анализе взаимодействия между модулями.

### Шаг 2: Консолидация результатов

AI-ассистент получает отчёты от агентов, анализирует их:
- Отфильтровывает ложные срабатывания
- Группирует найденные ошибки по критичности
- Определяет, какие ошибки требуют исправления

### Шаг 3: Исправление ошибок (если найдены)

Для каждой найденной ошибки:
1. AI-ассистент читает конкретный файл и строку, где обнаружена ошибка
2. Вносит исправление через `edit`
3. Если исправление затрагивает несколько файлов — вносит изменения во все затронутые файлы

### Шаг 4: Верификация

AI-ассистент запускает сборку на сервере (`www`):
```bash
ssh www "cd /home/kane/Projects/mh-compressor-manager/src/build && make -j4 2>&1 | tail -10"
```

Если сборка успешна — проверяет базовую работоспособность:
```bash
ssh www "sudo mh-compressor-manager --version"
```

### Шаг 5: Коммит и пуш (если были изменения)

Если были внесены исправления:
```bash
git add -A && git commit -m "fix: ..." && git push
```

### Шаг 6: Формирование отчёта

AI-ассистент выводит краткий отчёт:
- Количество проверенных файлов
- Количество найденных ошибок
- Количество исправленных ошибок
- Статус сборки (✅/❌)
- Предложение дальнейших действий

## Недостатки текущей логики

1. **Отсутствие единого прохода по ВСЕМ файлам** — агенты делят файлы между собой,
   но не проверяют их перекрёстно. Ошибка на стыке модулей может быть пропущена.
2. **Нет чёткого чек-листа** — каждый раунд может проверять разные аспекты.
3. **Нет проверки документации** — аудит касается только `src/`, но не `.md` файлов.
4. **Нет проверки тестов** — тестовые скрипты не анализируются.
5. **Нет фиксированного формата отчёта** — каждый раз отчёт выглядит по-разному.
6. **Нет сохранения истории** — результаты предыдущих аудитов не сравниваются.
7. **Агенты не имеют контекста предыдущих исправлений** — могут найти уже исправленную ошибку.

## Рекомендуемая оптимизация (для обсуждения)

1. **Единый последовательный проход** по всем файлам (один агент, все файлы).
2. **Фиксированный чек-лист** разделов кода (включая документацию и тесты).
3. **Формат отчёта** с таблицей: файл, строка, тип, критичность, статус.
4. **Сравнение с предыдущим раундом** — что нового, что осталось.
5. **Проверка сборки ДО начала аудита** — чтобы не искать ошибки в сломанном коде.
