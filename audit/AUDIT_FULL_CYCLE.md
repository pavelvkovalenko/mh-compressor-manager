# Процедура «Проведи полный аудит» — mh-compressor-manager

## Назначение

Данный файл фиксирует оптимизированную логику действий AI-ассистента при получении команды
**«проведи полный аудит»**.

**Приоритет:** Качество. Скорость — вторична.

---

## Архитектура процесса

### Принцип управляющего контекста

```
┌─────────────────────────────────────────────────────────────┐
│              УПРАВЛЯЮЩИЙ КОНТЕКСТ (Master)                   │
│  - Загружены ВСЕ файлы src/, docs/, tests/                   │
│  - Знает полную архитектуру и связи между модулями           │
│  - Строит граф зависимостей между модулями                   │
│  - Читает историю предыдущих аудитов                         │
│  - Принимает решение: ошибка это или предполагаемое поведение │
│  - Вносит изменения, не ломая другие части программы          │
│  - Обновляет CHANGELOG, ROUND_HISTORY, публикует отчёт       │
└───────────────────────┬─────────────────────────────────────┘
                        │
        ┌───────┬───────┼───────┬───────┐
        │       │       │       │       │
   ┌────▼──┐ ┌─▼────┐ ┌▼─────┐ ┌▼────┐ ┌▼──────┐
   │Субагент│ │Суба- │ │Суба- │ │Суба-│ │Суба-  │
   │A:Сжатие│ │гент B│ │гент C│ │гент D│ │гент E │
   │        │ │Монит.│ │I/O   │ │Опт.  │ │Безоп. │
   └───┬────┘ └──┬───┘ └──┬───┘ └──┬──┘ └──┬────┘
       │         │        │        │        │
       └─────────┴────────┴────────┴────────┘
                         │
               «Подозрение в ошибке»
               → Отчёт Master-у
               → Master верифицирует
               → Master исправляет
               → Master обновляет документацию
```

**Ключевое правило:** Субагенты ТОЛЬКО анализируют и сообщают о подозрениях.
Master принимает решение и вносит исправления.

## Пошаговая процедура

### Шаг 0: Pre-flight

**1. Проверка сборки:**
```bash
ssh www "cd /home/kane/Projects/mh-compressor-manager/src/build && make -j4 2>&1 | tail -5"
```
Если сборка сломана — аудит прекращается, сообщается о проблеме.

**2. Загрузка истории:**
- Прочитать `audit/ROUND_HISTORY.md` — знать что уже проверялось
- Прочитать `audit/CHANGELOG.md` — знать что уже исправлялось
- Прочитать последний отчёт из `audit/reports/` — знать метрики прошлого раунда
- Определить: это полный или инкрементальный аудит (см. раздел «Режимы»)

**3. Построение графа зависимостей:**
- Загрузить все `.h` и `.cpp` файлы
- Определить `#include` зависимости между модулями
- Построить таблицу «Затрагивает при изменении» (см. раздел выше)

**Формат отчёта субагента:**
```

  Обоснование: почему это может быть ошибкой
```

**Важно:** Субагент НЕ вносит изменений. Только сообщает подозрения.

### Шаг 3: Верификация Master-ом

Для КАЖДОГО подозрения от субагента:

1. **Master загружает полный контекст** затронутых файлов (если ещё не загружен)
2. **Master проверяет по истории** — эта ошибка уже исправлялась в раунде N?
3. **Master проверяет по графу зависимостей** — какие ещё модули затронет исправление?
4. **Master определяет:**
   - Это действительно ошибка? Или так задумано?
   - Если ошибка — какие ещё модули затронет исправление?
   - Есть ли побочные эффекты?
   - Какой раздел ТЗ нарушен?
5. **Master принимает решение:** исправить, игнорировать, или отложить

**Результат:** Отфильтрованный список подтверждённых ошибок с планом исправления.

### Шаг 4: Последовательное исправление

Master исправляет ошибки **ПО ОДНОЙ**, в порядке приоритета:

1. Исправить ошибку
2. Прочитать все файлы из графа зависимостей, которые затронуты исправлением
3. Убедиться что исправление не ломает другие модули
4. Проверить соответствие `.h` ↔ `.cpp` (сигнатуры, типы)
5. Обновить `audit/CHANGELOG.md` (если исправление значимое)
6. Только потом перейти к следующей ошибке

**Правило:** Одно исправление — одна проверка целостности — следующий.
Никаких массовых изменений без промежуточной проверки.

### Шаг 5: Автоматизированные проверки

После ручных исправлений Master запускает:

**5.1. Проверка конвенций (RULES.md):**
- [ ] `snake_case` функции, `PascalCase` классы, `m_` префикс
- [ ] `const` корректность
- [ ] Комментарии на русском языке
- [ ] `openat` с `O_NOFOLLOW` для защиты от symlink
- [ ] `lstat` не `stat`
- [ ] Нет `sprintf`, `strcpy`, `gets`
- [ ] `std::shared_lock` для чтения, `std::unique_lock` для записи

**5.2. Static Analysis Checklist:**
- [ ] **Use-after-free** — указатель используется после `free`/`delete`
- [ ] **Null pointer dereference** — разыменование без проверки на `nullptr`
- [ ] **Integer overflow** — `size_t` wrapping при арифметике
- [ ] **Signed/unsigned mismatch** — сравнение знаковых и беззнаковых
- [ ] **Buffer overread** — чтение за концом буфера

**5.3. Поиск мёртвого кода:**
- Неиспользуемые функции
- Неиспользуемые `#include`
- Deprecated функции, которые давно не используются (`compress_dual`)
- Недостижимый код (после `return`, `break`)

**5.4. Проверка конфигурации:**
- Все CLI аргументы документированы в `--help`
- Все ключи INI файла описаны в ТЗ
- Обратная совместимость — не ломаются ли старые конфиги

**5.5. Documentation Drift Detection:**
- Параметры CLI в `--help` соответствуют `config.cpp`
- Разделы ТЗ соответствуют реальному коду
- README.md отражает текущие возможности
- Примеры в документации работают

## Чек-листы для каждого субагента

### Для кода (.cpp/.h)
- [ ] Все `open()` имеют парный `close()` на всех путях выполнения
- [ ] Все `allocate()` имеют парный `free/release()` на всех путях
- [ ] `errno` сохраняется сразу после ошибки (`int err = errno`)
- [ ] Нет использования указателей после `free`/`delete`
- [ ] Нет гонок данных (shared mutable без mutex)
- [ ] Все ветки `if/else` возвращают значение
- [ ] Деструкторы корректно чистят ресурсы
- [ ] Нет переполнений (unsigned wraparound, integer overflow)
- [ ] API вызовы корректны (zlib, brotli, inotify, io_uring, systemd)
- [ ] Нет дубликатов `#include`
- [ ] `.h` и `.cpp` соответствуют друг другу (сигнатуры, типы)

### Для документации (.md)
- [ ] ТЗ соответствует коду (разделы 3.x, 4.x, 18.x)
- [ ] audit/AUDIT_CYCLE.md актуален
- [ ] audit/CHANGELOG.md отражает последние изменения
- [ ] audit/ROUND_HISTORY.md содержит последний раунд
- [ ] Нет противоречий между файлами
- [ ] README.md отражает текущее состояние проекта

### Для тестов (.sh)
- [ ] Скрипты имеют `set -e`
- [ ] Скрипты завершаются за разумное время (не таймаутят)
- [ ] Объём данных ≥ 1 ГБ (правило из TEST_SCRIPTS.md)
- [ ] Скрипты корректно убирают за собой
- [ ] Тесты проверяют критичные сценарии из ТЗ §18

### Для безопасности (дополнительно)
- [ ] Нет захардкоженных паролей, ключей, токенов
- [ ] `O_NOFOLLOW` для всех `open()`
- [ ] `lstat` вместо `stat` для проверки symlink
- [ ] `validate_filename()` вызывается для пользовательского ввода
- [ ] Seccomp sandbox настроен корректно
- [ ] Drop privileges работает (не root после инициализации)
- [ ] Rate limiter защищает от DoS
- [ ] CWE/OWASP чек-лист пройден

### Для производительности (дополнительно)
- [ ] Нет N+1 запросов (лишних системных вызовов в цикле)
- [ ] Нет избыточных аллокаций (переиспользование буферов)
- [ ] Нет лишних копирований (передача по ссылке/указателю)
- [ ] libdeflate используется если доступен
- [ ] CPU affinity настроен с циклическим распределением
- [ ] Cache-aware buffers корректно определяют L3

## Режимы аудита

### Полный аудит
- **Когда:** Запрошен явно командой «проведи полный аудит»
- **Глубина:** Все файлы, все разделы ТЗ, все субагенты (A-F), все шаги (0-13)
- **Время:** 30-60 мин
- **Результат:** Полная картина качества кода

### Инкрементальный аудит
- **Когда:** Прошло < 20 коммитов с прошлого полного аудита
- **Глубина:** Только изменённые файлы + критичные зависимости по графу
- **Субагенты:** Только те, чья область затронута изменениями
- **Шаги:** 0, 1 (частично), 2 (частично), 3, 4, 9, 12, 13
- **Время:** 10-20 мин
- **Результат:** Проверка что новые изменения не сломали существующее

**Правило переключения:** Если с последнего полного аудита прошло > 20 коммитов — запускать полный. Иначе — инкрементальный.

## Отличие от цикла аудита (AUDIT_CYCLE.md)

| Параметр | Цикл аудита | Полный аудит |
|----------|------------|-------------|
| **Глубина** | Быстрая проверка | Максимальная глубина |
| **Контекст** | Частичный (субагент) | Полный (Master) |
| **Граф зависимостей** | Не строится | Строится и используется |
| **Edge-Cases** | Не проверяются | Проверяются (Шаг 6) |
| **Static Analysis** | Не проверяется | Автоматический чек-лист (Шаг 5.2) |
| **CWE/OWASP** | Не проверяется | Фиксированный чек-лист (Шаг 7) |
| **Config Matrix** | Не проверяется | Все комбинации из ТЗ §18.4 (Шаг 8) |
| **Benchmark** | Не проводится | Before/After (Шаг 9) |
| **Valgrind** | Не запускается | Запускается если доступен (Шаг 9) |
| **Regress Tests** | Не создаются | Auto-generation (Шаг 10) |
| **Документация** | Не проверяется | Drift Detection (Шаг 5.5) |
| **Тесты** | Не проверяются | Проверяются |
| **Безопасность** | Не проверяется | Отдельный субагент E |
| **Мёртвый код** | Не ищется | Ищется автоматически (Шаг 5.3) |
| **Конвенции** | Не проверяются | Проверка по RULES.md (Шаг 5.1) |
| **История** | Не учитывается | Читается перед началом (Шаг 0) |
| **Changelog** | Не обновляется | Automated (Шаг 11) |
| **Отчёт** | Краткий | Форматированный + публикация (Шаг 13) |
| **Субагенты** | Анализируют и исправляют | Только анализируют, Master исправляет |
| **Время** | 10-20 мин | 30-60 мин |
