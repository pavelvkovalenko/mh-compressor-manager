# Процедура «Очисти мёртвый код и неиспользуемые зависимости»

## Назначение

Данный файл фиксирует оптимизированную логику действий AI-ассистента при получении команды
**«очисти мёртвый код и неиспользуемые зависимости»**.

**Приоритет:** Безопасность изменений. Скорость — вторична.
**Принцип:** Автоматический анализ компилятором → Отчёт → Последовательное удаление → Верификация.

## Пошаговая процедура

### Шаг 0: Pre-flight

1. **Проверка сборки:**
   ```bash
   ssh www "cd /home/kane/Projects/mh-compressor-manager/src/build && make -j4 2>&1 | tail -5"
   ```
   Если сборка сломана — очистка прекращается.

### Шаг 1: Автоматический анализ компилятором

AI запускает сборку со строгими флагами поиска мёртвого кода:
```bash
ssh www "cd /home/kane/Projects/mh-compressor-manager/src/build &&
  make clean &&
  cmake .. -DCMAKE_CXX_FLAGS=\"-Wall -Wextra -Wpedantic -Wunused-function -Wunused-variable -Wunused-parameter\" &&
  make -j4 2>&1 | grep -i 'warning: unused' | tee /tmp/dead_code_candidates.log"
```

**Результат:** Список кандидатов от компилятора.

### Шаг 2: AI Deep Scan (Ручная проверка)

Для кода, который компилятор мог пропустить (например, неиспользуемые функции в заголовках, dead CMake dependencies):

1. **Поиск неиспользуемых функций:**
   Для каждой функции, объявленной в `.h`:
   - `grep_search "<function_name>"` во всех `.cpp`.
   - Если вызовов нет (кроме объявления) → кандидат на удаление.

2. **Поиск неиспользуемых `#include`:**
   - Проверить, используются ли символы из этого заголовка в файле.
   - Если нет → кандидат на удаление.

3. **Проверка зависимостей CMake:**
   - Проверить `CMakeLists.txt` на наличие `find_package` / `pkg_check_modules`, библиотеки которых не линкуются или символы не используются.

4. **Построение графа зависимостей:**
   При удалении функции Master проверяет ВСЕ файлы, которые могут её вызывать (через `#include` цепочки).

### Шаг 3: Формирование Dry-run отчёта

Master формирует таблицу найденных проблем, отсортированную по уровню риска:

| Файл | Строка | Тип | Описание | Безопасность |
|------|--------|-----|----------|-------------|
| `compressor.cpp` | 690 | Unused Function | `compress_dual()` | ✅ Safe (deprecated, не вызывается) |
| `main.cpp` | 15 | Unused Include | `<unused_header.h>` | ✅ Safe |
| `threadpool.h` | 45 | Unused Variable | `m_debug_flag` | ✅ Safe |
| `CMakeLists.txt` | 50 | Unused Dep | `ZSTD` library | ⚠️ Medium (проверить conditional use) |

### Шаг 4: Последовательное удаление

Удаление производится **строго по приоритету** (см. Матрицу ниже):

1. **Safe:** Удаление без вопросов (unused includes, local vars).
2. **Medium:** Удаление с проверкой (unused private methods).
3. **Risky:** Удаление с подтверждением (unused public API).

**Правило:** Одно удаление → проверка сборки → следующее удаление.
Никаких массовых удалений без промежуточной проверки!

### Шаг 5: Верификация

После каждого блока удалений:
```bash
# Сборка
ssh www "cd /home/kane/Projects/mh-compressor-manager/src/build && make -j4 2>&1 | tail -10"

# Тесты (базовые)
ssh www "sudo mh-compressor-manager --version"
```

### Шаг 6: CI Gate Update (Prevention)

Чтобы мёртвый код не возвращался, AI предлагает добавить флаги в CI/Build скрипт:
```cmake
# В CMakeLists.txt или _local-build.sh
target_compile_options(mh-compressor-manager PRIVATE
    -Wall -Wextra -Wpedantic
    -Werror=unused-function   # Превращает warning в ошибку
    -Werror=unused-variable
)
```

## Матрица приоритетов

| Приоритет | Тип мёртвого кода | Пример | Действие |
|-----------|-------------------|--------|----------|
| **Safe** | Unused `#include` | `#include <vector>` (не используется) | Удалить сразу |
| **Safe** | Unused локальные переменные | `int x = 0;` | Удалить сразу |
| **Safe** | Закомментированный код | `/* void old_func() { ... } */` | Удалить сразу |
| **Medium** | Unused `private` методы | `void helper() {}` | Удалить после проверки |
| **Medium** | Unused параметры | `void func(int unused)` | `(void)unused;` или удалить |
| **Risky** | Unused `public` API | `void API::func()` | Проверить тесты/доки перед удалением |
| **Risky** | Unused CMake Deps | `find_package(ZSTD)` | Проверить `target_link_libraries` |

## Инкрементальный режим

**Когда:** Нужно проверить только изменения после последнего коммита.

**Процедура:**
1. Получить список изменённых файлов:
   ```bash
   git diff --name-only HEAD~N
   ```
2. Проверить только эти файлы + файлы, которые их включают (через `grep "#include"`).
3. Сфокусироваться на новых предупреждениях компилятора, появившихся после изменений.

## Отличие от полного аудита

| Параметр | Полный аудит | Очистка мёртвого кода |
|----------|-------------|-----------------------|
| **Цель** | Поиск багов и ошибок логики | Удаление лишнего кода и зависимостей |
| **Метод** | Анализ потоков, race conditions | Анализ ссылок (refs), unused symbols |
| **Инструменты** | Sub-agents, ручная проверка | Компилятор (`-Wunused`), `grep`, статический анализ |
| **Результат** | Исправленные баги | Удалённый код, упрощение кодовой базы |
| **Риск** | Средний (исправление логики) | Низкий (удаление неиспользуемого) |

## Чек-лист проверки

### Перед удалением
- [ ] Функция/переменная точно не используется (grep по всему проекту)
- [ ] Нет динамического вызова (через указатели на функции)
- [ ] Нет использования в тестах (проверить `tests/`)
- [ ] Нет использования в документации (проверить примеры в `.md`)

### После удаления
- [ ] Сборка проходит без ошибок
- [ ] Сборка проходит без новых предупреждений (в идеале)
- [ ] Базовые тесты проходят
- [ ] Размер бинарника не вырос
