# Процедура цикла аудита — mh-compressor-manager

## Режимы аудита

### Режим 1: Цикл аудита (стандартный)

```
ЦИКЛ:
  1. Запустить раунд аудита:
     - Проверить ВСЕ файлы .cpp/.h в src/
     - Если ошибок НЕ найдено → ВЫХОД из цикла
  2. Исправить ОДНУ из самых простых найденных проблем
     - Остальные ошибки НЕ трогать (сохранить консистентность)
  3. Запушить исправление
  4. Сбросить всю логику проверки (память о предыдущих проверках)
  5. Вернуться к пункту 1
```

### Режим 2: Цикл аудита с полным тестированием

```
ЦИКЛ:
  1. Запустить раунд аудита:
     - Проверить ВСЕ файлы .cpp/.h в src/ (полная проверка кода)
     - Если ошибок в коде НЕ найдено → перейти к пункту 2 (тесты)
     - ИНАЧЕ (найдены ошибки в коде):
       → Перейти к пункту 3 (исправление)
  2. Запустить тесты по ОДНОМУ:
     - Запустить один тест → проверить результат
     - Если тест выявил ошибку → перейти к пункту 3 (исправление)
     - Если тест пройден → запустить следующий тест
     - Если ВСЕ тестЫ пройдены → ВЫХОД из цикла
  3. Исправить ОДНУ из самых простых найденных проблем
     (ошибку в коде или в тесте — что проще исправить)
  4. Запушить исправление
  5. Сбросить всю логику проверки (память о предыдущих проверках)
  6. Вернуться к пункту 1 (НОВЫЙ РАУНД)
```

> **Важно:** Режим 2 = Режим 1 + серия тестов (не вместо, а в дополнение).
> Тесты запускаются только если код прошёл проверку без ошибок.
> При любой ошибке (в коде или в тесте) — исправляется одна проблема и цикл перезапускается.

### Режим 3: Полный аудит (код + тесты + документация)

```
ЦИКЛ:
  1. Запустить раунд аудита кода (как в Режиме 1):
     - Проверить ВСЕ файлы .cpp/.h в src/
     - Если ошибок НЕ найдено → перейти к пункту 2
     - ИНАЧЕ → перейти к пункту 4 (исправление)

  2. Запустить тесты (как в Режиме 2):
     - Запускать тесты по ОДНОМУ
     - Если ВСЕ тестЫ пройдены → перейти к пункту 3
     - ИНАЧЕ (тест провален) → перейти к пункту 4

  3. Проверить ВСЮ документацию проекта:
     A. Проверка внутренних ссылок (якорей):
        - Извлечь ВСЕ ссылки вида [#...] из файлов .md
        - Для каждой ссылки — найти соответствующий заголовок в документе
        - Формат якоря GitHub: #номер-текст-через-дефисы
          • Точки в нумерации УДАЛЯЮТСЯ (### 3.2.8 → #328-...)
          • Спецсимволы (:, (), — и т.д.) опускаются
          • Пробелы заменяются на дефисы
        - Пример: "### 3.2.8. Привязка потоков к ядрам CPU"
          → "#328-привязка-потоков-к-ядрам-cpu-cpu-affinity"
        - Отметить битые ссылки (якорь не соответствует заголовку)

     B. Проверка внешних ссылок (URL):
        - Извлечь ВСЕ ссылки вида [текст](https://...) из файлов .md
        - Проверить формат URL (должен начинаться с http:// или https://)
        - Отметить подозрительные ссылки:
          • Локальные пути (file://, /home/..., C:\...)
          • Временные URL (localhost, 127.0.0.1)
          • Ссылки на удалённые ресурсы (по возможности проверить через web_fetch)

     C. Проверка содержания (Table of Contents):
        - Если файл содержит раздел "Содержание"/"Contents"/"Оглавление":
          • Проверить что КАЖДЫЙ пункт содержания соответствует реальному заголовку
          • Проверить что ВСЕ заголовки верхнего уровня (##, ###) есть в содержании
          • Отметить отсутствующие пункты (раздел есть, но в содержании нет)
          • Отметить лишние пункты (в содержании есть, но раздела нет)

     D. Проверка на внутренние противоречия:
        - Проверить нумерацию разделов (нет ли двух разделов 8 и 9 в одном уровне)
        - Проверить перекрёстные ссылки (ссылка из раздела A ведёт на раздел B?)
        - Проверить дублирующие разделы (одинаковое название, разное содержимое)
        - Проверить консистентность терминов (одно и то же называется по-разному?)
        - Проверить версии (версия в README = версия в spec = версия в CMakeLists.txt?)

     E. Проверка формата и структуры:
        - Заголовки идут по порядку (## → ### → ####)
        - Нет пропущенных уровней (не ## сразу → ####)
        - Таблицы корректно отформатированы
        - Кодовые блоки закрыты (```...```)

     Если НЕ НАЙДЕНО ошибок в документации → ВЫХОД из цикла
     ИНАЧЕ (найдены ошибки в документации) → перейти к пункту 4

  4. Исправить ОДНУ из самых простых найденных проблем:
     (ошибку в коде, в тесте, ИЛИ в документации — что проще исправить)

  5. Запушить исправление

  6. Сбросить всю логику проверки (память о предыдущих проверках)

  7. Вернуться к пункту 1 (НОВЫЙ РАУНД)
```

> **Важно:** Режим 3 = Режим 2 + полная проверка документации.
> Включает проверку ссылок, содержания, противоречий, формата.
> При любой ошибке (код, тест, документация) — исправляется одна проблема и цикл перезапускается.

## Критерии выхода из цикла

- **Режим 1 (Аудит):** Все файлы src/ проверены, ошибок не найдено
- **Режим 2 (Тестирование):** Все файлы src/ проверены (без ошибок) И все тесты пройдены
- **Режим 3 (Полный аудит):** Код без ошибок + все тесты пройдены + вся документация корректна (ссылки, содержание, нет противоречий)

## Правила

1. **ОДНО исправление за раунд** — не массовые изменения
2. **Сброс контекста** — каждый раунд начинается с чистого листа
3. **Журналирование** — каждое изменение фиксируется в CHANGELOG.md
4. **Подсчёт раундов** — ведётся статистика количества раундов
