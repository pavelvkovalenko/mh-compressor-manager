# Процедура «Проверь документацию» — mh-compressor-manager

## Назначение

Перед выполнением любого действия (аудит, тестирование, исправление кода, изменение кода, пуш) —
проверить, не изменились ли файлы документации и настроек с момента их последней загрузки в контекст беседы.

## Обоснование

ТЗ, спецификации, требования и настройки проекта могут быть изменены **вне текущего контекста беседы**:

| Источник изменений | Пример |
|-------------------|--------|
| **Вручную пользователем** | Пользователь отредактировал ТЗ, добавил новые требования |
| **Третьим лицом** | Другой разработчик, менеджер, ревьюер внёс изменения |
| **В другой ветке/PR** | Изменения в другой ветке обсуждения или Pull Request |
| **Через файлы .md** | Любое изменение файлов документации вне текущего сеанса |
| **Через .settings** | Изменение настроек проекта |

**Риск:** Выполнение действий на основании устаревших требований приведёт к:
- Созданию кода, не соответствующего актуальным требованиям
- Нарушению логики, определённой в обновлённом ТЗ
- Массовым переделкам и ошибкам

## Механизм отслеживания

### Принцип работы

**Метка времени хранится в КОНТЕКСТЕ БЕСЕДЫ (в памяти AI-ассистента), а НЕ в файле.**

Это позволяет запускать **параллельные ветки обсуждения** без конфликтов — каждая ветка хранит свою собственную метку времени.

### Скрипт проверки

```bash
# Проверить изменения (Linux/macOS) — метка передаётся как аргумент
bash audit/check_context_changes.sh <timestamp>

# Вывести список отслеживаемых файлов с датами
bash audit/check_context_changes.sh --list-files
```

```cmd
REM Проверить изменения (Windows)
audit\check_context_changes.bat <timestamp>

REM Список файлов
audit\check_context_changes.bat --list-files
```

### Формат вывода скрипта

**Если изменений нет:**
```
📅 Текущее время: 2026-04-11 17:00:00
📅 Документация загружена в контекст: 2026-04-11 15:30:00

✅ Документация не изменялась с момента загрузки в контекст
```

**Если есть изменения:**
```
📅 Текущее время: 2026-04-11 17:00:00
📅 Документация загружена в контекст: 2026-04-11 15:30:00

⚠️  ОБНАРУЖЕНЫ ИЗМЕНЕНИЯ В ДОКУМЕНТАЦИИ (2 файлов):

   🔴 docs/specification/TECHNICAL_SPECIFICATION.md (изменён: 2026-04-11 16:45:00, 1 ч назад)
   🔴 .settings (изменён: 2026-04-11 16:50:00, 55 мин назад)

📌 Действия:
   1. Прочитай изменённые файлы: cat <файл>
   2. Загрузи изменения в контекст беседы
   3. Обнови метку времени в контексте (не в файле!)

💡 Текущий timestamp для сохранения в контекст: 1712858400
```

## Процедура

### Шаг 0: Запуск скрипта проверкики

Получить текущий timestamp:
```bash
date +%s  # Linux/macOS: например 1712858400
```

Передать метку из контекста беседы в скрипт:
```bash
bash audit/check_context_changes.sh 1712858400
```

**Если скрипт обнаружил изменения** → перейди к Шагу 1.
**Если изменений нет** → перейди к Шагу 3.

### Шаг 1: Чтение изменённых файлов

Для каждого файла из списка изменений:
```bash
cat <путь_к_файлу>
```

**Обязательные файлы для проверки:**
- `docs/specification/TECHNICAL_SPECIFICATION.md` — ТЗ и требования
- `docs/development/QWEN.md` — правила и контекст AI-ассистента
- `docs/development/RULES.md` — правила написания кода
- `docs/development/DEPLOY.md` — сборка и деплой
- `CONTRIBUTING.md` — Pull Request и коммиты
- `README.md` — обзор проекта
- `.settings` — настройки проекта (если существует)
- `audit/CHANGELOG.md` — журнал изменений
- `audit/ROUND_HISTORY.md` — история раундов аудита
- `tests/TEST_SCRIPTS.md` — реестр тестовых скриптов

### Шаг 2: Первичная загрузка в контекст

Если метка времени ещё не установлена в контексте беседы:

1. Прочитай **все** файлы из списка обязательных файлов
2. Зафиксируй текущее состояние документации в контексте беседы
3. Получи и запомни текущий timestamp:
   ```bash
   date +%s  # → например, 1712858400
   ```
4. Запомни в контексте: `context_doc_timestamp=1712858400`
5. Перейди к Шагу 3.

### Шаг 3: Анализ изменений

Для каждого изменённого файла определи:
1. **Характер изменений:**
   - Новые требования к функциональности
   - Изменения в логике поведения
   - Обновлённые настройки проекта
   - Новые правила написания кода
   - Исправления ошибок в документации
2. **Влияние на текущую задачу:**
   - Требует ли изменения плана действий
   - Нужно ли пересмотреть предыдущие решения

### Шаг 4: Уведомление пользователя

Если обнаружены изменения, выведи сообщение:

```
📄 Обнаружены изменения в документации:
- Файл: TECHNICAL_SPECIFICATION.md (изменены требования к сжатию)
- Файл: QWEN.md (добавлено правило проверки документации)

Изменения учтены. Продолжаю работу с обновлёнными требованиями.
```

### Шаг 5: Обновление метки времени в контексте

После загрузки изменений в контекст беседы:
```bash
date +%s  # Получить новый timestamp
```
Обнови `context_doc_timestamp` в контексте беседы на новое значение.

> ⚠️ **НЕ записывай timestamp в файл!** Метка хранится только в контексте беседы.

## Критерии завершения

- [ ] Скрипт `check_context_changes.sh` запущен с меткой из контекста
- [ ] Если есть изменения — все изменённые файлы прочитаны
- [ ] Новые требования учтены в плане действий
- [ ] Пользователь уведомлён об изменениях (если есть)
- [ ] Метка времени обновлена в контексте беседы (если документация загружена)

## Когда вызывать процедуру

| Ситуация | Вызывать? |
|----------|-----------|
| Перед циклом аудита | ✅ Да |
| Перед тестированием | ✅ Да |
| Перед исправлением кода | ✅ Да |
| После пуша изменений | ✅ Да |
| После паузы в работе (более 5 минут) | ✅ Да |
| При создании новой ветки обсуждения | ✅ Да |
| При явном запросе пользователя «проверь документацию» | ✅ Да |
| В середине выполнения задачи (без изменений кода) | ❌ Нет |
