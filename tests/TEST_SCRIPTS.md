# Реестр тестовых скриптов

Этот файл ведёт учёт всех тестовых скриптов для повторного использования при тестировании mh-compressor-manager.

## Расположение
Все скрипты хранятся в папке `tests/scripts/`.

## Правило: Минимальный объём данных для нагрузочного тестирования

> ⚠️ **Объём тестовых данных при нагрузочном тестировании НЕ ДОЛЖЕН быть менее 1 ГБ.**
>
> **Обоснование:** Меньший объём данных не выявляет проблемы производительности, утечки памяти,
> деградацию при длительной работе и некорректную работу с большими файлами.
> Уменьшение объёма тестируемых данных **недопустимо** — это превращает нагрузочное
> тестирование в функциональное, что скрывает реальные проблемы.
>
> **Требования:**
> - Минимальный объём: **1 ГБ**
> - Минимальное количество файлов: **300**
> - Предварительно сжатые файлы: до **50%** от общего количества
> - Типы файлов: HTML, CSS, JS, JSON, TXT, XML, SVG
> - Размеры файлов: от 256 байт до 100 МБ (разнообразие)

## Правило использования
При необходимости выполнить скрипт на удалённой системе (`www`):
1. Создать/изменить скрипт **локально** в `tests/scripts/`
2. Скопировать на сервер: `scp tests/scripts/script.sh www:/tmp/script.sh`
3. Выполнить: `ssh www "bash /tmp/script.sh"`
4. Сохранить финальную версию в репозиторий
5. Обновить этот файл

## Скрипты

### 1. gen_loadtest.sh
- **Назначение:** Генерация нагрузочных тестовых данных (1GB+)
- **Тип:** Генерация данных
- **Статус:** ✅ Готов к использованию
- **Описание:** Создаёт 6 типов файлов (html, css, js, json, xml, txt) в `/srv/123/loadtest/` с предварительным сжатием ~40% файлов для проверки механизма stale.
- **Параметры:**
  - HTML: 80 файлов (~9MB)
  - CSS: 50 файлов (~7MB)
  - JS: 50 файлов (~6MB)
  - JSON: 60 файлов (~49MB)
  - XML: 50 файлов (~20MB)
  - TXT: 80 файлов логов (~1.7GB)
  - Итого: ~1.8GB, 370 файлов
- **Последнее использование:** 2026-04-11, нагрузочное тестирование памяти и initial scan
- **Файл:** `tests/scripts/gen_loadtest.sh`
- **Связь со спецификацией:** Раздел 18.2.1 (тестовые данные)

### 2. loadtest_comprehensive.sh
- **Назначение:** Комплексное нагрузочное тестирование (генерация + сжатие + валидация)
- **Тип:** Нагрузочное + интеграционное тестирование
- **Статус:** ✅ Готов к использованию
- **Описание:** 7 этапов: подготовка → генерация 500 файлов (~346MB) → stale detection (40%) → запуск сервиса с мониторингом → проверка корректности gzip/brotli → стресс-тест (100 файлов за раз) → финальные метрики → очистка
- **Этапы:**
  1. Остановка сервиса, очистка директории
  2. Генерация HTML/CSS/JS/JSON/TXT файлов
  3. Предварительное сжатие 40% gzip + установка старых mtime
  4. Запуск сервиса, мониторинг каждые 30 сек (memory, CPU, прогресс)
  5. Валидация: gzip -t, brotli -t, сравнение размеров распаковки
  6. Стресс-тест: 100 новых файлов за 1 секунду
  7. Итоговая таблица + очистка
- **Результаты первого запуска (2026-04-11):**
  - 500 файлов сгенерировано, 346MB
  - 914 сжатых файлов создано
  - gzip/brotli: 20/20 валидных, 0 ошибок
  - Память: peak 346MB → 8.6MB (утечки нет)
  - 0 реальных ошибок
- **Файл:** `tests/scripts/loadtest_comprehensive.sh`
- **Связь со спецификацией:** Раздел 18.2.2 (нагрузочный тест)

### 3. test_stale_cleanup.sh
- **Назначение:** Тестирование очистки stale-файлов (STALE-1—STALE-6)
- **Тип:** Функциональное + интеграционное тестирование
- **Статус:** ✅ Создан (2026-04-12)
- **Описание:** Проверка корректного удаления stale-сжатых файлов при уменьшении исходника ниже порога.
- **Тест-кейсы:**
  - **STALE-1:** Initial scan: файл 1025 байт → сжат → уменьшен до 171 байт externally → stale удалён
  - **STALE-2:** Runtime: файл 500 байт сжат → модифицирован до 200 байт → stale удалён
  - **STALE-3:** Файл 300 байт сжат → заменён файлом 100 байт с тем же mtime → stale удалён
  - **STALE-4:** 50 файлов стали ниже порога одновременно → все stale удалены
  - **STALE-5:** Файл 200 байт (ниже порога), stale существует → stale удалён
  - **STALE-6:** Файл ниже порога, stale нет → без действий
- **Файл:** `tests/scripts/test_stale_cleanup.sh`
- **Связь со спецификацией:** Раздел 18.6.3 (STALE-1—STALE-6)

### 5. test_race_condition.sh (⏳ Требуется создать)
- **Назначение:** Тестирование race condition при изменении файла во время сжатия (RACE-1—RACE-17)
- **Тип:** Стресс-тестирование + интеграционное тестирование
- **Статус:** 🔴 Требуется реализация
- **Описание:** Проверка корректности при удалении/изменении/перемещении файла во время его сжатия.
- **Тест-кейсы:**
  - **RACE-1—7:** Обычные файлы (<1 МБ) — удаление, изменение, переименование, уменьшение ниже порога
  - **RACE-8—13:** Большие файлы (10 МБ+) — те же сценарии
  - **RACE-14—17:** Параллельные race conditions — одновременные изменения нескольких файлов
- **Метод:** Искусственная задержка сжатия (замедление чтения/записи) для гарантированного попадания в окно race condition.
- **Файл:** `tests/scripts/test_race_condition.sh`
- **Связь со спецификацией:** Раздел 18.6.6 (RACE-1—RACE-17)

### 6. test_atomicity.sh (⏳ Требуется создать)
- **Назначение:** Тестирование атомарности записи (ATOM-1—ATOM-4)
- **Тип:** Интеграционное + стресс-тестирование
- **Статус:** 🔴 Требуется реализация
- **Описание:** Проверка атомарной записи сжатых файлов (`.tmp` → `rename()`), отсутствия полузаписанных файлов.
- **Тест-кейсы:**
  - **ATOM-1:** Сбой записи (нет места на диске) → `.tmp` удалён
  - **ATOM-2:** Crash программы во время записи → `.gz.tmp` удалён при restart
  - **ATOM-3:** nginx запрашивает `.gz` в момент `rename()` → старый или новый, не полузаписанный
  - **ATOM-4:** Два потока сжимают один файл → только один завершает
- **Файл:** `tests/scripts/test_atomicity.sh`
- **Связь со спецификацией:** Раздел 18.6.6 (ATOM-1—ATOM-4)

### 7. test_orphan_check.sh (⏳ Требуется создать)
- **Назначение:** Проверка отсутствия orphan-файлов после всех операций (ORPHAN-1—ORPHAN-3)
- **Тип:** Валидационное тестирование
- **Статус:** 🔴 Требуется реализация
- **Описание:** Поиск `.gz`/`.br`/`.tmp` без соответствующего оригинала.
- **Тест-кейсы:**
  - **ORPHAN-1:** После 100 race condition операций — нет `.gz`/`.br` без оригинала
  - **ORPHAN-2:** После 100 race condition операций — нет `.tmp` файлов
  - **ORPHAN-3:** После серии перемещений папок — нет stale на старых путях
- **Метод:** Рекурсивный обход monitored-директории, сравнение списков оригиналов и сжатых файлов.
- **Файл:** `tests/scripts/test_orphan_check.sh`
- **Связь со спецификацией:** Раздел 18.6.6 (ORPHAN-1—ORPHAN-3)

- **Тип:** Параметрическое тестирование
- **Статус:** 🔴 Требуется реализация

  - **CFG-4:** Оба алгоритма, среднее сжатие
  - **CFG-5:** Оба алгоритма, min сжатие
  - **CFG-6:** Только gzip, level 6 (по умолчанию)
- **Проверки для каждой конфигурации:** запуск, создание правильных расширений, валидность сжатия, min_compress_size, debounce, идемпотентность, удаление.

### 11. test_memory_adaptive.sh (⏳ Требуется создать)
- **Назначение:** Тестирование адаптивного управления памятью
- **Тип:** Нагрузочное + параметрическое тестирование
- **Статус:** 🔴 Требуется реализация
- **Описание:** Сравнение 4 стратегий управления памятью пулов.
- **Стратегии для сравнения:**
  - **A:** Немедленное освобождение (после каждого файла)
  - **B:** Периодическое освобождение (таймер, 5 мин)
  - **C:** Освобождение при превышении лимита
  - **D:** Гибридная (таймер + лимит)
- **Метрики:** Общее время, CPU usage, пиковая память, количество аллокаций/деаллокаций.
- **Результат:** Таблица сравнения, рекомендуемая стратегия, обоснование.
- **Файл:** `tests/scripts/test_memory_adaptive.sh`
- **Связь со спецификацией:** Раздел 18.3.2 (сравнение стратегий)

### 12. test_lifecycle_full.sh (⏳ Требуется создать)
- **Назначение:** Полная проверка lifecycle приложения с перемещениями
- **Тип:** End-to-end интеграционное тестирование
- **Статус:** 🔴 Требуется реализация
- **Описание:** Сценарий из 10 шагов: создание → переименование → перемещение → остановка → external changes → запуск → проверка.
- **Шаги:**
  1. Создать 50 файлов, запустить → все сжаты
  2. Переименовать 10 файлов внутри → stale удалены, новые сжаты
  3. Переместить 10 наружу → stale удалены
  4. Переместить 10 внутрь → новые сжаты
  5. Переместить папку наружу → stale внутри удалены
  6. Переместить папку внутрь → все файлы обработаны
  7. Остановить программу
  8. Externally: переименовать, переместить, удалить
  9. Запустить → stale удалены, новые сжаты
  10. Проверить отсутствие orphan-файлов
- **Файл:** `tests/scripts/test_lifecycle_full.sh`
- **Связь со спецификацией:** Раздел 18.6.5 (Lifecycle-сценарий)
