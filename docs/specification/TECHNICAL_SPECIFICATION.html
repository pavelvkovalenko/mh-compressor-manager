2. Назначение и принципы работы

✅ Основная цель: Автоматизация процесса сжатия статических файлов для оптимизации работы веб-сервера.

2.1. Мониторинг

Программа отслеживает изменения в файловой системе в режиме реального времени (мониторинг).

2.2. Обработка файлов

При обнаружении новых или измененных файлов с подходящими расширениями программа создает их сжатые копии в той же директории.

2.3. Алгоритмы сжатия

Алгоритм Расширение Библиотека
Gzip .gz zlib
Brotli .br libbrotli

2.4. Сохранность данных

Исходные файлы не удаляются и не модифицируются.

2.5. Метаданные

Сжатые копии должны наследовать от оригинала: - Время изменения (mtime) — обязательно (критично для HTTP-кэширования) - Права доступа (permissions) — обязательно - Контекст безопасности SELinux — обязательно (если применимо) - Владельца и группу (UID/GID) — обязательно

Время создания (birth time) сжатого файла должно соответствовать реальному текущему времени его создания (не наследуется от оригинала). Это полезно для аудита и отслеживания того, когда именно было выполнено сжатие.

2.6. Архитектура

Компонент Назначение
Основной процесс Демон, работащий в фоне
Поток-диспетчер Обрабатывает события файловой системы (inotify)
Пул потоков Выполняет операции сжатия асинхронно
Systemd-интеграция Взаимодействие через уведомления о готовности (sd_notify)

3. Требования к функциональности

3.1. Мониторинг файловой системы

3.1.1. Рекурсивное сканирование

Программа должна рекурсивно сканировать указанные целевые директорторию(ии).

3.1.2. Механизм отслеживания

Используется механизм inotify (API или библиотека) для отслеживания событий: - Создание файлов - Модификация файлов - Удаление файлов

3.1.3. Обработка событий

Событие Действие
Создание/Изменение Если файл имеет подходящее расширение, планируется задача на сжатие
Удаление Если удаляется исходный файл, соответствующие сжатые копии (.gz, .br) также должны быть удалены
Игнорирование Программа не должна следовать за символическими ссылками (symlinks)
Фильтрация Не обрабатывать файлы, которые уже являются сжатыми копиями (оканчиваются на .gz или .br)

3.1.4. Debounce (Задержка)

📌 Требование: Для предотвращения множественной обработки одного файла при частых изменениях (например, при копировании), сжатие выполняется только после того, как файл не изменялся в течение заданного интервала (параметр debounce_delay).

3.1.5. Идемпотентность

Если сжатый файл уже существует и его время модификации новее или равно времени модификации исходного файла, повторное сжатие не производится (при первоначальном сканировании).

3.1.6. Очистка устаревших (stale) сжатых файлов

Проблема: Возможна ситуация, при которой исходный файл был больше минимального порога сжатия, но затем уменьшился и стал ниже порога. При этом содержимое файла изменилось, возможно даже в то время, когда программа не работала. В результате сжатый файл содержит данные, не соответствующие текущему содержимому оригинала, хотя время модификации может совпадать (если файл был изменён externally и затем восстановлен с теми же метаданными).

Пример сценария:

Шаг 1: file.txt = 1025 байт  →  сжат в file.txt.gz (150 байт)
Шаг 2: file.txt изменён externally, стал 171 байт (< порога 256)
Шаг 3: Программа запускается, видит:
       - file.txt: 171 байт < 256 → НЕ сжимает
       - file.txt.gz: 150 байт, mtime = старый
       - Содержимое file.txt.gz НЕ соответствует file.txt! ← ПРОБЛЕМА!

Требование: При первоначальном сканировании и при обработке событий программа должна:

  1. Обнаруживать stale-файлы: Если исходный файл существует, но его размер ниже минимального порога сжатия (min_compress_size), при этом сжатые копии (.gz, .br) существуют — сжатые копии считаются недействительными и подлежат удалению.

  2. Проверять соответствие содержимого: Даже если mtime исходного файла совпадает с mtime сжатого файла, но размер исходного файла ниже порога — сжатый файл должен быть удалён, поскольку его содержимое может не соответствовать оригиналу.

  3. Алгоритм проверки при сканировании:

Для каждого найденного исходного файла:
  ЕСЛИ file.size < min_compress_size:
    ЕСЛИ file.gz ИЛИ file.br существуют:
      УДАЛИТЬ file.gz И file.br
      ЛОГИРОВАТЬ: "Удалён stale-сжатый файл (оригинал ниже порога)"
    ПРОДОЛЖИТЬ (не сжимать)
  ИНАЧЕ:
    Выполнить стандартную stale-check по mtime
  1. Алгоритм при событии изменения:
При событии модификации file.txt:
  ЕСЛИ file.size < min_compress_size:
    ЕСЛИ file.gz ИЛИ file.br существуют:
      УДАЛИТЬ file.gz И file.br
      ЛОГИРОВАТЬ: "Удалён stale-сжатый файл после модификации"
    ПРОДОЛЖИТЬ (не сжимать)
  ИНАЧЕ:
    Запланировать сжатие
  1. Логирование: Удаление stale-файлов из-за того что оригинал стал ниже порога, должно логироваться на уровне INFO:
[INFO] Удалён stale-сжатый файл /srv/www/file.txt.gz: исходный файл 171 байт < порога 256 байт

3.1.7. Обработка перемещений (move/rename) файлов и папок

Проблема: Механизм inotify генерирует пары событий IN_MOVED_FROM + IN_MOVED_TO при перемещении/переименовании файлов внутри monitored-директории и между директориями. Однако: - Если файл перемещён из monitored-директории — приходит только IN_MOVED_FROM (или IN_DELETE) - Если файл перемещён в monitored-директорию — приходит только IN_MOVED_TO (или IN_CREATE) - При переименовании внутри monitored-директории — приходит пара IN_MOVED_FROM + IN_MOVED_TO с одинаковым cookie - Если программа не работала в момент перемещения — stale-файлы остаются на старом месте - При перемещении папки — inotify может не отследить рекурсивно все файлы внутри

Требования к обработке перемещений:

  1. Переименование файла внутри monitored-директории:
  2. Обнаружить пару IN_MOVED_FROM → IN_MOVED_TO (по cookie)
  3. Удалить старые сжатые копии со старого пути
  4. Создать новые сжатые копии на новом пути (если файл выше порога)

  5. Если файл ниже порога — только удалить старые копии

  6. Перемещение файла ИЗ monitored-директории:

  7. При событии IN_MOVED_FROM или IN_DELETE: удалить сжатые копии (.gz, .br)

  8. Не пытаться сжать файл (его больше нет в monitored-зоне)

  9. Перемещение файла В monitored-директорию:

  10. При событии IN_MOVED_TO или IN_CREATE: обработать как новый файл
  11. Если файл выше порога — сжать

  12. Если файл ниже порога — не сжимать, проверить наличие stale-копий

  13. Перемещение папки внутри monitored-директории:

  14. При перемещении папки — все файлы внутри неё должны быть переобработаны
  15. Старые сжатые копии в исходной папке — удалены

  16. Новые файлы на новом месте — обработаны (сжаты или пропущены по размеру)

  17. Перемещение папки ИЗ monitored-директории:

  18. Все сжатые копии внутри перемещённой папки должны быть удалены

  19. Рекурсивный обход: удалить все .gz и .br в перемещённой папке

  20. Перемещение папки В monitored-директорию:

  21. Рекурсивное сканирование перемещённой папки
  22. Обработка всех подходящих файлов (сжатие или пропуск по размеру)

  23. Добавление новых inotify-watch на поддиректории

  24. Перемещение при неработающей программе:

  25. При запуске (initial scan) обнаружить stale-файлы на старых путях и удалить их
  26. Обнаружить файлы на новых путях и обработать их

  27. Проверка: stale-файл существует, но соответствующего исходного файла нет → удалить

  28. Корректность cookie для пары MOVED_FROM/MOVED_TO:

  29. События с одинаковым cookie относятся к одному перемещению
  30. Если cookie не совпадает или одно событие из пары потеряно — обработать оба события независимо (как удаление + создание)

3.1.8. Корректность при изменении файла во время сжатия (Race Condition)

Проблема: В многопоточной системе между моментом постановки файла в очередь на сжатие и моментом завершения записи сжатого файла исходный файл может быть удалён, перемещён или изменён. Это особенно критично для больших файлов (10 МБ+), время сжатия которых может составлять секунды.

Возможные сценарии race condition:

Сценарий Что происходит Проблема
Удаление во время сжатия Файл поставлен в очередь → пользователь удалил файл → поток сжатия всё ещё читает/сжимает Чтение удалённого файла, запись stale-копии
Изменение во время сжатия Файл поставлен в очередь → пользователь измени файл → поток сжатия читает старые данные → записывает устаревшую копию Содержимое .gz/.br не соответствует текущему файлу
Перемещение во время сжатия Файл поставлен в очередь → пользователь переместил файл → поток сжатия пишет на старый путь Orphan-файл на старом пути, нет копии на новом
Изменение размера ниже порога Файл поставлен в очередь → файл уменьшен до <256 байт → поток сжатия всё ещё пытается сжать Нарушение политики min_compress_size

Требования к корректной обработке:

  1. Валидация перед чтением: Перед началом чтения файла для сжатия поток обязан проверить:
  2. Файл всё ещё существует
  3. Размер файла ≥ min_compress_size
  4. mtime файла совпадает с mtime на момент постановки в очередь (или новее)

  5. Файл находится в том же пути, что и на момент постановки в очередь

  6. Валидация перед записью: Перед записью сжатого файла проверить:

  7. Исходный файл всё ещё существует
  8. mtime исходного файла не изменился с момента начала чтения

  9. Если проверка не пройдена — отменить сжатие, удалить временный файл, логировать на уровне DEBUG

  10. Отмена при получении события изменения: Если во время сжатия поступает новое событие IN_MODIFY для того же файла:

  11. Текущее сжатие не отменять (довести до конца)
  12. Запланировать повторное сжатие после завершения текущего

  13. Это предотвращает бесконечный цикл отмен

  14. Обработка удаления во время сжатия:

  15. Если файл удалён во время чтения — прервать чтение, записать WARNING в лог
  16. Если файл удалён во время записи — дописать временный файл, затем удалить его (не оставлять полузаписанным)

  17. Частичные файлы обязаны удаляться: unlink(output.c_str());

  18. Обработка перемещения во время сжатия:

  19. Если файл перемещён — дописать сжатый файл на текущий путь (не на старый)
  20. Если путь больше не в monitored-зоне — удалить временный файл

  21. Проверить, нужно ли сжать файл на новом месте (может быть ниже порога)

  22. Атомарная запись: Запись сжатого файла должна быть атомарной:

  23. Сначала писать во временный файл (.gz.tmp, .br.tmp)
  24. После успешной записи — переименовать (rename()) в целевой
  25. При любой ошибке — удалить временный файл

  26. Это гарантирует что nginx никогда не прочитает полузаписанный .gz/.br

  27. Логирование race condition:

[DEBUG] Race condition для /srv/www/large.log: файл изменён во время сжатия, результат отменён
[DEBUG] Race condition для /srv/www/large.log: файл удалён во время чтения, сжатие прервано
[INFO] Повторное сжатие /srv/www/large.log запланировано после завершения текущего
  1. Защита для больших файлов (>10 МБ):
  2. Перед началом сжатия сохранить stat-информацию (размер, mtime, inode)
  3. После завершения сжатия — повторно вызвать stat() и сравнить
  4. Если inode измениён — файл был перемещён/заменён, отменить результат
  5. Если mtime новее — файл изменён, отменить результат
  6. Если размер < min_compress_size — удалить результат, записать DEBUG

3.2. Алгоритмы сжатия

3.2.1. Уровень сжатия

Уровень сжатия для Gzip и Brotli должен быть настраиваемым.

3.2.2. Многопоточность

Поддержка многопоточности: параллельное сжатие разных файлов через пул потоков.

3.2.3. Ограничение нагрузки

Количество параллельных операций сжатия не должно превышать заданное значение (параметр threads).

3.2.4. Однократное чтение исходного файла

Требование: Исходный файл должен быть прочитан ровно один раз перед началом сжатия, независимо от количества алгоритмов сжатия (gzip, brotli, или оба одновременно).

Алгоритм:

1. Прочитать исходный файл в память (буфер)
2. Сжать буфер через gzip → записать .gz
3. Сжать тот же буфер через brotli → записать .br
4. Освободить буфер

Требования к реализации: - Буфер в памяти должен содержать полное содержимое файла - Размер буфера = размер исходного файла - Оба алгоритма сжатия работают с одной и той же копией данных в памяти - Передача данных в алгоритмы — по указателю (const uint8_t*), без копирования буфера - Если сжатие одного алгоритма завершилось ошибкой — буфер всё ещё валиден для второго алгоритма - После завершения всех операций сжатия буфер обязан быть освобождён

Пример (псевдокод):

// НЕПРАВИЛЬНО: файл читается дважды
compress_gzip(path);   // read(path) → compress → write
compress_brotli(path); // read(path) → compress → write ← повторное чтение!

// ПРАВИЛЬНО: файл читается один раз, оба алгоритма из одного буфера
std::vector<uint8_t> data = read_file_to_memory(path);
compress_gzip_from_memory(data.data(), data.size());   // compress → write
compress_brotli_from_memory(data.data(), data.size()); // compress → write

Ограничение по памяти: Для файлов размером более доступной свободной памяти: - Использовать потоковое чтение чанками (например, по 4 МБ) - Чанк читается один раз, передаётся обоим алгоритмам, затем следующий чанк - Общий объём памяти не должен превышать 4 * max_chunk_size

3.2.5. Архитектура пула потоков: один поток — один файл целиком

Требование: Каждый поток в пуле обрабатывает файл полностью и целиком: чтение → сжатие gzip → сжатие brotli → запись → освобождение. Не допускается разделение одного файла между несколькими потоками или создание отдельных потоков для каждого алгоритма сжатия.

Модель обработки:

Поток 1 (ядро 0): файл_A → read → gzip → brotli → free → файл_B → ...
Поток 2 (ядро 1): файл_C → read → gzip → brotli → free → файл_D → ...
Поток 3 (ядро 2): файл_E → read → gzip → brotli → free → файл_F → ...

Запрещённые модели:

❌ НЕПРАВИЛЬНО (2 потока на один файл):
  Поток 1: файл_A → read → gzip → write
  Поток 2: файл_A → read → brotli → write  ← двойное чтение, 2 ядра

❌ НЕПРАВИЛЬНО (разделение файла):
  Поток 1: первая половина файла_A → gzip
  Поток 2: вторая половина файла_A → brotli ← потеря контекста сжатия

Обоснование: см. раздел 19. Обоснование требований к оптимизации скорости сжатия.

3.2.6. Переиспользование буфера между файлами

Требование: Буфер для чтения исходных файлов должен переиспользоваться между задачами сжатия в рамках одного потока, а не перевыделяться для каждого файла.

Требования к реализации: - Каждый поток пула владеет собственным буфером (thread_local или член класса worker) - Буфер увеличивается (reserve) только при необходимости, но не уменьшается - Перевыделение (malloc/realloc) происходит только если текущий файл больше capacity буфера - После завершения сжатия буфер очищается (clear()), но память не освобождается (capacity сохраняется)

Пример (псевдокод):

// НЕПРАВИЛЬНО: перевыделение для каждого файла
for (auto& file : files) {
    std::vector<uint8_t> buf(file.size());  // malloc каждый раз!
    read_and_compress(file, buf);
}

// ПРАВИЛЬНО: один буфер переиспользуется
class WorkerThread {
    std::vector<uint8_t> m_buffer;  // переиспользуется между файлами

    void process_file(const fs::path& path) {
        if (path.file_size() > m_buffer.capacity()) {
            m_buffer.reserve(path.file_size());  // realloc только при необходимости
        }
        read_file_to_buffer(path, m_buffer);
        compress(m_buffer.data(), path.file_size());
        m_buffer.clear();  // данные сброшены, память сохранена
    }
};

3.2.7. Использование libdeflate для gzip (рекомендуется)

Требование: При наличии библиотеки libdeflate программа должна использовать её вместо стандартной zlib для сжатия gzip.

Обоснование: libdeflate обеспечивает 2x более высокую скорость сжатия по сравнению с zlib при том же уровне сжатия (уровень 6). Библиотека является drop-in заменой и полностью совместима с форматом .gz.

Библиотека Скорость (level 6) Относительная производительность
zlib ~300 МБ/сек baseline (1.0x)
libdeflate ~600 МБ/сек 2.0x
zlib-ng ~350 МБ/сек 1.2x

Требования к реализации: - При сборке: если libdeflate доступен — использовать его, иначе fallback на zlib - Формат выходного файла .gz идентичен независимо от библиотеки - API сжатия: libdeflate_gzip_compress() вместо compress2()

3.2.8. Привязка потоков к ядрам CPU (CPU affinity)

Требование: Каждый поток в пуле должен быть привязан к отдельному ядру CPU через sched_setaffinity().

Требования к реализации: - При создании потока пула — определить его порядковый номер (0, 1, 2, ...) - Вызвать sched_setaffinity() с маской, содержащей одно соответствующее ядро - Если количество потоков превышает количество физических ядер — использовать циклическое распределение - При невозможности привязки (нет прав, контейнер) — продолжить работу без привязки, записать WARNING в лог

Пример (псевдокод):

void WorkerThread::run() {
    // Привязка к ядру
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(m_thread_id % std::thread::hardware_concurrency(), &cpuset);
    sched_setaffinity(0, sizeof(cpuset), &cpuset);

    // Основной цикл обработки
    while (running) {
        auto task = queue.pop();
        process_file(task);
    }
}

Обоснование: см. раздел 19. Обоснование требований.

3.2.9. Адаптивный размер буфера на основе размера кэша CPU

Требование: При запуске программа должна определить размер кэша L3 процессора и адаптировать размер буфера чтения/сжатия для максимальной эффективности использования кэш-памяти.

Обоснование: Размеры кэшей L3 на серверных процессорах варьируются от 12 МБ (consumer) до 480 МБ (Xeon Granite Rapids) — разница в 40 раз. Фиксированный размер буфера (например, 256 КБ) не использует преимущества больших кэшей на мощных серверах и может превышать кэш на слабых машинах, вызывая вытеснение данных в RAM (в 100-200 раз медленнее L1).

Алгоритм определения размера кэша:

  1. Основной метод — sysfs:
  2. Путь: /sys/devices/system/cpu/cpu0/cache/index<N>/
  3. Для каждого индекса прочитать level, type, size, shared_cpu_list
  4. Найти кэш с level = 3 (L3) и type = Unified

  5. Распарсить size (формат: 33554432 или 32M)

  6. Fallback методы:

  7. Если sysfs недоступен (контейнер, WSL) — использовать эвристику: 2 МБ

  8. sysconf(_SC_LEVEL*_CACHE_SIZE) — ненадёжен для L3 на Linux

  9. Расчёт оптимального буфера: buffer_size = L3_cache_size / thread_count buffer_size = clamp(buffer_size, 64 КБ, 16 МБ)

  10. Минимум: 64 КБ (покрывает типичные веб-файлы, помещается в L2)

  11. Максимум: 16 МБ (избегание вытеснения из L3 на системах с малым кэшем)

  12. Для файлов меньше буфера:

  13. Читать весь файл целиком (один read, один буфер)
  14. Оба алгоритма (gzip + brotli) сжимают из одного буфера без копирования

Пример (псевдокод):

class CacheInfo {
    static CacheInfo detect() {
        size_t l3 = detect_l3_cache_size();  // из sysfs
        size_t threads = std::thread::hardware_concurrency();
        size_t per_thread = l3 / threads;
        return {
            .l3_size = l3,
            .optimal_buffer = clamp(per_thread, 64 * 1024, 16 * 1024 * 1024)
        };
    }

    static size_t detect_l3_cache_size() {
        std::ifstream f("/sys/devices/system/cpu/cpu0/cache/index3/size");
        if (f.is_open()) {
            std::string line; std::getline(f, line);
            return parse_size_string(line);  // "32M" → 33554432
        }
        return 2 * 1024 * 1024;  // fallback: 2 МБ
    }
};

Требования к логированию: - При старте: записать INFO с размером L3 и вычисленным размером буфера - При fallback: записать WARNING о невозможности определения кэша

[INFO] CPU cache: L3 = 32 МБ, потоков = 4, буфер = 8 МБ
[WARNING] Не удалось определить размер кэша CPU, используется fallback: 2 МБ

4. Требования к минимальному размеру файла для сжатия

📋 Основание: Анализ индустриальных практик и технических исследований (см. раздел 15. Обоснование требований).

4.1. Пороговое значение

Параметр Значение Обоснование
MIN_COMPRESS_SIZE 256 байт Файлы <150 байт могут увеличиться после сжатия; 256 байт — консенсус индустрии (nginx gzip_min_length 256)

4.2. Стратегия сжатия по размеру файла

Размер файла Действие
< 256 байт Не сжимать — файлы могут увеличиться после сжатия
≥ 256 байт Сжимать — предварительное сжатие с максимальным уровнем даёт положительный эффект для всех файлов ≥ 256 байт

Обоснование: Поскольку mh-compressor-manager выполняет сжатие предварительно (не на лету) с максимальной степенью сжатия (gzip level 9, brotli level 11), затраты процессора не являются критичными. Файлы в диапазоне 256–1023 байт также сжимаются, поскольку дают экономию трафика 30-50%.

4.3. Требования к логированию малых файлов

При пропуске файлов размером менее 256 байт система должна: - Логировать событие на уровне DEBUG (не WARNING и не ERROR) - Указывать причину: file size below minimum threshold (256 bytes) - Не создавать предупреждения для пользователя

Пример:

[DEBUG] Пропуск файла /srv/www/small.js: размер 128 байт < минимального порога (256 байт)

4.4. Требования к конфигурации

Параметр настраивается через конфигурационный файл:

[compression]
min_compress_size = 256

Требования: - Значение по умолчанию: 256 байт - Минимально допустимое значение min_compress_size: 256 байт - При задании значения < 256 — используется значение по умолчанию (256) с логированием предупреждения - При некорректном значении (нечисловое, отрицательное) — используется значение по умолчанию с логированием предупреждения

Обоснование порога 256 байт: Файлы <150 байт после сжатия gzip/brotli могут стать больше оригинала из-за накладных расходов на служебные заголовки (18 байт gzip) и словари Хаффмана (~50-65 байт). Значение 256 байт — консенсус индустрии (nginx gzip_min_length 256, brotli_min_length 256), обеспечивающий стабильное сжатие с экономией 30-70% трафика при файлах ≥256 байт.

4.5. Проверка размера файла

Псевдокод:

bool should_compress(size_t file_size) {
    if (file_size < m_cfg.min_compress_size) {
        return false;
    }
    return true;
}

Требования: - Проверка выполняется до начала сжатия - Проверка выполняется после открытия файла и получения его размера - Не пытаться сжимать файлы ниже порога

4.6. Статистика пропущенных файлов

Система должна вести счётчики: - files_skipped_small — количество пропущенных файлов (размер < порога) - bytes_skipped_small — общий объём пропущенных данных (байт)

Эти счётчики должны быть доступны в: - Выводе статуса сервиса - Метриках (если реализовано) - Логах статистики при завершении работы

4.7. Предварительное сжатие в фоне

Контекст: mh-compressor-manager выполняет сжатие предварительно (не на лету) с использованием максимальной степени сжатия: - gzip: уровень 9 - brotli: уровень 11

Вывод: Затраты процессора на сжатие не являются критичными, так как выполняются в фоне. Однако распаковка на клиенте остаётся фактором, поэтому порог 256 байт сохраняет актуальность.

4.8. Совместимость с nginx

Настройки mh-compressor-manager должны быть совместимы с рекомендациями nginx: - gzip_min_length 256; - brotli_min_length 256;

Это обеспечивает единообразие обработки файлов на обоих уровнях.

15.2. Технические накладные расходы

GZIP overhead

  • Заголовок: 10 байт
  • Футер (CRC32 + размер): 8 байт
  • Итого служебных данных: 18 байт
  • Словари Хаффмана для малых файлов: ~50–65 байт
  • Общий overhead: 18–83 байта в зависимости от содержимого

Риск увеличения файла

  • Файлы <150 байт после сжатия gzip могут стать больше оригинала
  • Причина: служебные заголовки + словарь сжатия превышают выигрыш от алгоритма DEFLATE
  • Механизм предотвращения: DEFLATE автоматически формирует блоки типа 0 (Uncompressed), когда сжатие не даёт выгоды

Сетевые соображения

  • Один TCP-пакет: файлы до ~860 байт помещаются в один пакет (MTU ~1500 байт с учётом заголовков)
  • Вывод Akamai: сжатие файлов, помещающихся в один пакет, не даёт заметных сетевых преимуществ

15.5. Почему выбран порог 256 байт для mh-compressor-manager

✅ Предварительное сжатие (не на лету) — затраты на процессор не важны
✅ Максимальная степень сжатия — brotli level 11 эффективнее даже для малых файлов
✅ Распаковка на клиенте — дешёвая операция, но для файлов <256 байт время распаковки сравнимо с экономией трафика
✅ Сетевые пакеты — файлы до ~860 байт помещаются в один TCP-пакет, но 256+ байт уже дают экономию 30–70%
✅ Совместимость с nginx — согласуется с настройками по умолчанию (gzip_min_length 256;, brotli_min_length 256;)

17. Обоснование требований к временным меткам файлов

17.1. Исследование: какие временные метки использует nginx

Типы временных меток файлов в Linux

Метка Описание Используется nginx?
mtime (modification time) Время последнего изменения содержимого файла ✅ ДА — основная
ctime (change time) Время изменения метаданных (права, владелец) ❌ Нет
atime (access time) Время последнего чтения файла ❌ Нет
birth time (creation time) Время создания файла (inode) ❌ Нет

Как nginx генерирует HTTP-заголовки для статических файлов

Last-Modified:

// Исходный код nginx
r->headers_out.last_modified_time = of.mtime;

Используется исключительно mtime файла.

ETag:

ETag = "%xT-%xO"  →  "62b1fab3-267"
         ↑     ↑
      mtime  Content-Length

ETag формируется из mtime + размер файла (в шестнадцатеричном формате).

Механизм отдачи файлов nginx

Диск → stat() → проверка mtime → sendfile() → сеть
                  ↑
          Last-Modified + ETag

nginx использует системный вызов stat(), который считывает mtime, но не birth time. Директива sendfile передаёт файл напрямую из дискового кэша ядра в сетевой сокет — время создания не участвует.

17.2. Выводы исследования

Время создания (birth time) НЕ влияет на отдачу nginx/CDN, поскольку: 1. nginx не передаёт birth time в HTTP-ответах 2. CDN-провайдеры (Cloudflare, Akamai и др.) кэшируют файлы на основе HTTP-заголовков (Cache-Control, ETag, Last-Modified), а не файловых метаданных 3. Механизм sendfile не использует birth time

Время изменения (mtime) КРИТИЧНО, поскольку: 1. Определяет заголовок Last-Modified в HTTP-ответе 2. Участвует в генерации ETag 3. Влияет на кэширование в браузерах и CDN 4. Определяет валидность кэша при ревалидации (If-Modified-Since)

17.3. Требования к обработке временных меток

mtime — наследовать от оригинала

Параметр Требование Причина
mtime сжатого файла Должен совпадать с mtime оригинала nginx использует mtime для Last-Modified и ETag. Если сжатый файл будет иметь другой mtime, nginx отдаст другие кэш-заголовки, что приведёт к некорректному кэшированию и сбросу кэша у пользователей

Пример проблемы при неправильном mtime:

Оригинал style.css:  mtime = 2026-04-10 12:00:00
Сжатый style.css.br:  mtime = 2026-04-11 08:00:00  ← НЕВЕРНО!

nginx отдаёт для style.css:     Last-Modified: Sat, 10 Apr 2026 12:00:00 GMT
nginx отдаёт для style.css.br:  Last-Modified: Sun, 11 Apr 2026 08:00:00 GMT  ← РАЗНЫЕ!

Результат: браузер не может корректно кэшировать пару файлов

birth time — не наследовать, устанавливать текущее

Параметр Требование Причина
birth time сжатого файла Текущее реальное время создания Полезно для аудита: администратор может определить, когда именно было выполнено сжатие. Не влияет на HTTP-ответы nginx

Пример для аудита:

$ stat style.css style.css.br

  File: style.css
    Birth: 2026-04-01 10:00:00  ← оригинал создан ранее
    Modify: 2026-04-10 12:00:00

  File: style.css.br
    Birth: 2026-04-11 08:30:15  ← сжатие выполнено сегодня
    Modify: 2026-04-10 12:00:00  ← mtime совпадает с оригиналом

17.4. Приоритеты наследования метаданных

Метаданные Критичность Причина
mtime 🔴 Критично Определяет Last-Modified и ETag — основу HTTP-кэширования
permissions 🟡 Важно Безопасность доступа к файлам
owner/group 🟡 Важно Совместимость с nginx user
SELinux context 🟡 Важно (если применимо) Доступ на SELinux-системах
birth time 🟢 Не критично для HTTP Не влияет на отдачу nginx, полезно только для аудита

18.3. Тестирование адаптивного управления памятью

18.3.1. Требования к поведению пулов памяти

Во время всплеска нагрузки: - Если свободная память на устройстве позволяет — система должна увеличить лимиты пулов памяти для достижения максимальной производительности - Увеличение лимитов происходит автоматически при обнаружении свободной памяти - Программа должна логировать факт увеличения лимитов (уровень INFO)

После падения нагрузки: - Размер буферного пула обязан уменьшиться до базового значения - Уменьшение происходит автоматически после завершения обработки очереди - Программа должна логировать факт уменьшения лимитов (уровень INFO)

Оптимизация при большой очереди: - Если в очереди на сжатие много файлов — НЕ освобождать пул после каждого сжатого файла - Причина: затраты на выделение/освобождение памяти снижают общую производительность - Сжатие пулов должно происходить периодически или при превышении лимита: - Вариант A: по таймеру (например, раз в 5 минут) - Вариант B: при превышении максимального размера пула - Вариант C: при отсутствии задач в очереди в течение N секунд

18.3.2. Сравнительное тестирование стратегий управления памятью

Требование: Провести сравнение производительности следующих стратегий:

Стратегия Описание Измеряемые метрики
A. Немедленное освобождение Освобождение буфера после каждого сжатого файла Общее время, CPU, память
B. Периодическое освобождение (таймер) Освобождение раз в N минут (рекомендуется: 5 мин) Общее время, CPU, память
C. Освобождение при превышении лимита Освобождение только при достижении максимального размера пула Общее время, CPU, память
D. Гибридная (таймер + лимит) Освобождение при превышении лимита ИЛИ раз в N минут Общее время, CPU, память

Результат сравнения должен быть задокументирован в описании проекта с указанием: - Рекомендуемой стратегии - Обоснования выбора - Измеренных значений для каждой стратегии - Сценариев, при которых каждая стратегия наиболее эффективна

⚠️ Важно: Результаты сравнения необходимо сохранить в документации проекта, чтобы в будущем случайно не удалить функционал, который необходим для повышения эффективности программы.

19.1. Исследование: контекстные переключения и их влияние на производительность

Накладные расходы на переключение контекста

Метрика Значение Источник
Контекстный переключ (CPU-affinity) 1.2–1.5 мкс Eli Bendersky, 2018 [1]
Контекстный переключ (без affinity) ~2.2 мкс Eli Bendersky, 2018 [1]
Память на поток (стек) 8 МБ виртуальной Eli Bendersky, 2018 [1]
Резидентная память на поток (RSS, активный) ~400 КБ Eli Bendersky, 2018 [1]

Вывод: Контекстный переключ стоит ~1500 нс — это эквивалентно 150-1500 промахам L3-кэша. При миграции потока между ядрами данные из L1/L2 вытесняются, что добавляет ещё больше задержек.

Штраф за промах кэша

Уровень Задержка Относительная стоимость
L1 hit ~1 нс (4 такта) baseline (1x)
L2 hit ~3 нс (12 тактов) 3x
L3 hit ~10-20 нс (40-75 тактов) 10-20x
RAM access (miss всех уровней) ~100-200 нс 100-200x

Вывод: При миграции потока между ядрами данные L1/L2 теряются. Для compute-bound задач (сжатие) это означает повторную загрузку данных из RAM — в 100-200 раз медленнее L1.

19.2. Исследование: характер нагрузки сжатия

Библиотека Характер Однопоточная скорость
zlib (gzip, level 6) CPU-bound ~300 МБ/сек
libdeflate (gzip, level 6) CPU-bound ~600 МБ/сек (2x zlib)
brotli (level 11) CPU-bound ~5-15 МБ/сек
brotli (level 6) CPU-bound ~30-50 МБ/сек

Источники: - libdeflate vs zlib: encode.su forums [2] - zlib-ng discussion #871 [3] - Hacker News discussion on single-threaded compression [4]

Вывод: Сжатие — вычислительно-ёмкая (CPU-bound) задача, не I/O-bound. Данные из памяти читаются быстрее, чем сжимаются. Основной лимит — CPU cycles, не диск.

19.3. Сравнение архитектур обработки файла

Архитектура A: 2 потока на один файл (НЕ рекомендуется)

Поток 1 (ядро A): read(file) → compress_gzip → write(.gz)
Поток 2 (ядро B): read(file) → compress_brotli → write(.br)
Параметр Значение Проблема
Чтений с диска 2 Двойная нагрузка на I/O
Копирований в память 2 Двойная нагрузка на RAM bandwidth
Занятых ядер 2 Меньше файлов обрабатывается параллельно
Контекстных переключений N + M Межпоточная синхронизация
Cache thrashing Высокий Два ядра конкурируют за кэш

Итог: 2 ядра заняты одним файлом. При 1000 файлов и 4 ядрах — только 4 файла обрабатываются одновременно, но каждое занимает 2 потока.

Архитектура B: 1 поток — 1 файл целиком (рекомендуется)

Поток 1 (ядро 0): файл_A → read → gzip → brotli → free → файл_B → ...
Поток 2 (ядро 1): файл_C → read → gzip → brotli → free → файл_D → ...
Параметр Значение Преимущество
Чтений с диска 1 ✅ Минимальная I/O нагрузка
Копирований в память 1 ✅ Минимальная RAM bandwidth
Занятых ядер 1 на файл ✅ Больше файлов параллельно
Контекстных переключений N ✅ Меньше sync overhead
Cache locality Высокая ✅ Данные остаются в L3 между gzip и brotli

Итог: 4 ядра = 4 файла одновременно. Каждый файл обрабатывается полностью одним потоком.

Сравнение для мелких файлов (<1 МБ)

Метрика 2 потока/файл 1 поток/файл
Время на файл T_read + T_gzip + T_brotli + overhead T_read + T_gzip + T_brotli
Overhead context switch ~3-5 мкс (2 переключ.) ~1.5 мкс (0 переключ.)
Пропускная способность N/2 файлов в единицу времени N файлов

Для мелких файлов 1 поток/файл даёт ~2x пропускную способность за счёт отсутствия дублирования I/O и контекстных переключений.

Сравнение для больших файлов (>10 МБ)

Метрика 2 потока/файл 1 поток/файл
Время на файл max(T_gzip, T_brotli) T_gzip + T_brotli
Теоретический выигрыш Быстрее (параллельное сжатие) Медленнее (последовательное)
Реальный выигрыш Минимальный (brotli level 11 — bottleneck) —

Нюанс: Для больших файлов brotli level 11 работает значительно медленнее gzip (5-15 МБ/сек vs 300-600 МБ/сек). Даже при параллельном сжатии общее время определяется brotli. Поэтому выигрыш от 2-поточной архитектуры для больших файлов незначителен по сравнению с потерями от двойного чтения.

19.4. Исследование: переиспользование буфера

Стоимость malloc/free

Операция Задержка Источник
malloc + free (малый блок) ~100-500 нс Общие бенчмарки
malloc + free (большой блок, >4 КБ) ~1-5 мкс (syscall при необходимости) glibc malloc internals
reserve без увеличения capacity ~0 нс std::vector specification

При обработке 10 000 файлов: - Без переиспользования: 10 000 × malloc + 10 000 × free = 1-50 мс только на аллокации - С переиспользованием: 1 × malloc (initial reserve) = ~1 мкс

Выигрыш: до 50x меньше времени на аллокации.

Cache behavior при переиспользовании буфера

Сценарий L3 cache hit rate Причина
Новый malloc каждый файл Низкий Данные разбросаны по памяти
Переиспользование буфера Высокий Тот же регион памяти, данные «тёплые» в кэше

Для файлов < 1 МБ (типичная веб-статика) буфер помещается в L3 cache (~12-24 МБ). При переиспользовании данные остаются в кэше между gzip и brotli — ~10-20% ускорение второго алгоритма.

19.6. Исследование: CPU affinity (привязка к ядру)

Преимущества

Преимущество Эффект
Нет cache thrashing при миграции Данные L1/L2 сохраняются между итерациями
Предсказуемая производительность Нет «провалов» при миграции
Меньше context switch overhead 1.2-1.5 мкс вместо 2.2 мкс [1]

Недостатки

Недостаток Риск
Планировщик OS не балансирует нагрузку Если ядро загружено другим процессом — поток ждёт
Неравномерная нагрузка Простой других ядер при обработке одного большого файла

Источник: sched_setaffinity man page [6], StackOverflow discussion on CPU affinity [7]

Вывод: CPU affinity даёт 5-10% выигрыш для compute-bound workload при условии, что другие процессы не конкурируют за те же ядра. Рекомендуется использовать с пулом потоков, где каждый поток привязан к своему ядру.

struct CacheInfo {
    size_t l1_dcache_per_core;   // L1 data cache на ядро (обычно 32-48 КБ)
    size_t l2_per_core;          // L2 cache на ядро (обычно 512 КБ – 2 МБ)
    size_t l3_total;             // L3 cache общий (12-480 МБ)
    size_t l3_shared_cores;      // Сколько ядер делят L3
    size_t thread_count;         // std::thread::hardware_concurrency()

     * @brief Оптимальный размер буфера на один поток
     * L3 / thread_count, clamp(64 КБ, 16 МБ)

     * @brief Оптимальный размер чанка для streaming
     * То же что optimal_buffer_size(), но минимум 64 КБ
     */
    size_t optimal_chunk_size() const;

     * Основной метод: sysfs
     * Fallback: 2 МБ L3

Алгоритм определения L3 через sysfs:

1. Для каждого CPU (cpu0, cpu1, ...):
   a. Открыть /sys/devices/system/cpu/cpu<N>/cache/
   b. Для каждого index (index0, index1, ...):
      - Прочитать level      → ищем level=3
      - Прочитать type       → ищем "Unified"
      - Прочитать size       → парсим "32M" → 33554432
      - Прочитать shared_cpu_list → дедупликация (один L3 на несколько ядер)
   c. Сохранить максимальный L3

2. Если sysfs недоступен (контейнер, WSL):
   - Fallback: L3 = 2 МБ

3. Расчёт:
   - optimal_buffer = L3 / thread_count
   - optimal_buffer = clamp(optimal_buffer, 64KB, 16MB)

Задача 2.3: Streaming чанковое сжатие для больших файлов

Проблема: Для файлов > optimal_chunk_size() (например, 10 МБ файл при буфере 3 МБ) данные не помещаются в L3 целиком. При naivной one-shot обработке:

read(10 МБ) → gzip читает из RAM (L3 miss) → brotli читает из RAM (L3 miss)
= 2 чтения из RAM в кэш

Решение: Чанковая streaming обработка:

for chunk in file:
    read(chunk) → buffer (3 МБ, помещается в L3)
    → gzip(chunk)   — данные в L3 (cache hit)
    → brotli(chunk) — данные ЕЩЁ В L3 (cache hit!)
    → следующий чанк (предыдущий вытесняется — это OK)

Ключевой принцип: gzip и brotli обрабатывают один и тот же чанк пока он в L3. Каждое чтение из RAM происходит только один раз.

Streaming API для gzip (zlib):

size_t chunk_size = worker.m_optimal_chunk_size;  // 3 МБ

    // Мелкий файл — one-shot (данные помещаются в L3)
    uint8_t* buf = worker.m_buffer.data();
    read_file(fd, buf, file_size);
    compress_gzip_from_memory(buf, file_size, ...);
    compress_brotli_from_memory(buf, file_size, ...);
} else {
    // Большой файл — streaming чанками
    GzipStreamState gz_state;
    BrotliStreamState br_state;

    gzip_stream_start(gz_state, cfg.gzip_level, gz_path);
    brotli_stream_start(br_state, cfg.brotli_level, br_path);

    uint8_t* buf = worker.m_buffer.data();
    for (size_t offset = 0; offset < file_size; offset += chunk_size) {
        size_t this_chunk = std::min(chunk_size, file_size - offset);
        read_file_chunk(fd, buf, offset, this_chunk);

        // Оба алгоритма обрабатывают один чанк — данные в L3
        gzip_stream_process(gz_state, buf, this_chunk);
        brotli_stream_process(br_state, buf, this_chunk);
    }

    gzip_stream_finish(gz_state);
    brotli_stream_finish(br_state);
}

22. Локализация (i18n)

22.1. Архитектура

Программа использует GNU gettext для локализации пользовательских сообщений (логи, --help, диагностика). Подход идентичен GNU coreutils (ls, df) и systemd.

Принцип: Исходный код содержит только английские строки. Переводы хранятся в отдельных .po файлах и компилируются в бинарные .mo файлы. При отсутствии .mo файла программа не падает — возвращает оригинальные английские строки.

    _("File %s"), path); ──xgettext→ ru.po  ← перевод  ──msgfmt→ /usr/share/locale/ru/
                                                              LC_MESSAGES/mh-compressor-manager.mo

Макрос Назначение Пример
_(str) Перевод строки через dgettext() _("File compressed")
_fmt(fmt, ...) Перевод + printf-форматирование _fmt("File: %s, size: %zu", path, size)
N_(str) Пометка строки без перевода (статические массивы) N_("%b %e %Y")

22.4. Форматирование сообщений

Запрещено: std::format(_(str), args) — несовместимо с GCC 15 (требует constexpr формат-строку).

Разрешено: printf-style через vsnprintf в Logger::*_fmt():

// ✅ ПРАВИЛЬНО (gettext-совместимо, GCC 14/15)
Logger::info_fmt(_("Compressed %s: %zu bytes"), path.c_str(), size);

// ❌ НЕПРАВИЛЬНО (не компилируется на GCC 15)
Logger::info(std::format(_("Compressed %s", path), size));

Спецификаторы формата: | Тип | Спецификатор | Пример | |-----|-------------|--------| | std::string | %s + .c_str() | path.c_str() | | size_t | %zu | file_size | | int | %d | level | | long | %ld | count | | errno | %s + strerror(errno) | strerror(errno) |

22.10. Полный цикл управления переводами

Архитектура перевода:

Исходный код (.cpp/.h)    translations/                    Установленные файлы
  Logger::info(_("..."))  ├── mh-compressor-manager.pot    /usr/bin/mh-compressor-manager
        │                 ├── ru.po                        /usr/share/locale/ru/
        ├── xgettext ────▶│   (переводчик заполняет)       │   LC_MESSAGES/
        │                 └── ru.mo ──msgfmt──▶             │       mh-compressor-manager.mo
        └── компиляция                                     └── ...

Полный цикл добавления/обновления перевода:

  1. Генерация шаблона .pot: bash # xgettext сканирует все _("...") и N_("...") в исходниках xgettext --from-code=UTF-8 --keyword=_ --keyword=N_ \ --output=translations/mh-compressor-manager.pot \ src/*.cpp src/*.h

  2. Обновление .po из .pot: bash # msgmerge добавляет новые строки, обновляет позиции, помечает obsolete msgmerge --update --no-fuzzy-matching translations/ru.po translations/mh-compressor-manager.pot

  3. Очистка obsolete строк (опционально): bash # Удаляет строки, которых больше нет в исходном коде msgattrib --no-obsolete -o translations/ru.po translations/ru.po

  4. Перевод новых строк: Редактор открывает translations/ru.po и заполняет msgstr для пустых строк: ```po # Было (после msgmerge): msgid "Starting mh-compressor-manager" msgstr ""

# Стало (после перевода): msgid "Starting mh-compressor-manager" msgstr "Запуск mh-compressor-manager" ```

  1. Компиляция .po → .mo: bash msgfmt translations/ru.po -o translations/ru.mo

  2. Установка .mo в систему: bash sudo cp translations/ru.mo /usr/share/locale/ru/LC_MESSAGES/mh-compressor-manager.mo

22.11. Ключевые правила перевода

Правило Пример Почему

printf-style формат %s, %d, %zu vsnprintf не понимает {}
.c_str() для std::string path.string().c_str() %s ожидает const char*
