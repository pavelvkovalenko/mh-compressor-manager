# Qwen Code - Проект mh-compressor-manager

## О проекте
Автоматическое сжатие статического контента для веб-сервера nginx с целью снижения сетевого трафика и ускорения загрузки страниц.

**Язык:** C++23
**Сборка:** CMake
**Платформа:** Linux (Fedora/RHEL)

## Цикл аудита
Процедура аудита определена в [audit/AUDIT_CYCLE.md](../../audit/AUDIT_CYCLE.md).

**Два режима:**
1. **Стандартный**: аудит → исправление одной ошибки → пуш → сброс → повтор
2. **С полным тестированием**: аудит → тесты → исправление → пуш → сброс → повтор

**Правила:**
- Одно исправление за раунд (не массовые изменения)
- Сброс контекста между раундами
- Журналирование в audit/CHANGELOG.md
- Подсчёт раундов в audit/ROUND_HISTORY.md

## Предложение действий после операций

После завершения любой операции (аудит, тестирование, исправление кода, изменение кода, пуш) —
в конце вывода предложи пользователю список дальнейших действий.

**Формат:** Один блок кода со всеми пунктами (компактно, строка выделяется мышкой и копируется через Ctrl+C):

```
🔄 1) Проведи цикл аудита
🧪 2) Проведи цикл аудита с полным тестированием
📚 3) Проведи полный аудит (код + тесты + документация)
🚀 4) Проведи нагрузочное тестирование
🛡️ 5) Проведи аудит безопасности
📈 6) Проведи бенчмарк производительности
📝 7) Подготовь отчёт для Changelog
📦 8) Собери RPM-пакет и протестируй установку
🧹 9) Очисти мёртвый код
```

> ⚠️ **ВАЖНО:** Все пункты в ОДНОМ блоке кода. На основе контекста AI может предложить альтернативные варианты.

### Быстрый выбор по номеру

Если пользователь отправил **только номер** пункта (например, `1`, `2`, `3` и т.д.) —
проверь, показывался ли список рекомендуемых действий **непосредственно перед этим сообщением**.

**Если список показывался** — выполни действие, соответствующее этому номеру из списка.

**Если список НЕ показывался** — воспринимай цифру как обычное число, НЕ связанное со списком действий.
Не пытайся интерпретировать её как выбор пункта.

**Пример (список показан):**
```
🔄 1) Проведи цикл аудита
🧪 2) Проведи цикл аудита с полным тестированием
✅ 3) Проведи тестирование
```
Пользователь: `2` → AI: запускает цикл аудита с полным тестированием.

**Пример (список НЕ показан):**
Пользователь: `3` → AI: воспринимает как обычное число (например, количество файлов, версию и т.д.).

**Правила:**
- Номер из списка работает ТОЛЬКО если список показан непосредственно перед сообщением пользователя
- Если номер вне диапазона — уточни у пользователя
- Если пользователь написал номер с текстом (например, `2, но с фокусом на gzip`) — выполни действие с учётом уточнения

### Правила предложения действий:

1. **Всегда предлагай** после: аудита, тестирования, исправления кода, пуша, любого изменения кода
2. **Формат:** каждый пункт должен быть отдельной строкой с эмодзи-маркером
3. **Копирование:** текст должен быть кликабельным (markdown ссылка или блок кода)
4. **Количество:** 5-8 вариантов, от наиболее вероятных к менее вероятным
5. **Контекст:** предлагай действия уместные для текущей ситуации:
   - После аудита → тестирование, нагрузочное тестирование
   - После исправления → новый раунд аудита, тестирование
   - После тестов → аудит, пуш, подготовка PR
   - После пуша → аудит нового кода, тестирование на сервере
6. **Альтернативы:** На основе контекста и предыдущих действий AI-ассистент может предложить свои варианты действий вместо фиксированного пункта.

## Правило: Перед любым действием — проверь документацию

**Перед выполнением ЛЮБОГО действия** (аудит, тестирование, исправление кода, изменение кода, пуш) —
сначала выполни процедуру «Проверь документацию» (см. [audit/CHECK_DOCUMENTATION.md](../../audit/CHECK_DOCUMENTATION.md)).

**Причина:** ТЗ, спецификации, требования и настройки проекта могут быть изменены:
- Вручную пользователем
- Третьим лицом (другим разработчиком, менеджером)
- В другой ветке обсуждения или Pull Request
- Через изменение файлов `.md` вне текущего контекста беседы

Если изменения обнаружены — **в первую очередь учитывай обновлённые требования** и загрузи их в контекст текущей беседы.

### Использование скрипта

```bash
# Проверить изменения (Linux/macOS) — метка из контекста беседы
bash audit/check_context_changes.sh <timestamp>

# Вывести список отслеживаемых файлов
bash audit/check_context_changes.sh --list-files

# Windows
audit\check_context_changes.bat <timestamp>
```

> ⚠️ **Метка времени хранится в КОНТЕКСТЕ БЕСЕДЫ, а НЕ в файле.**
> Каждая ветка обсуждения хранит свою собственную метку — конфликтов нет.
>
> **После загрузки документации в контекст** — получи текущий timestamp (`date +%s`)
> и запомни его в контексте как `context_doc_timestamp`.

## Правило: Обязательная актуализация спецификации при обновлении версии

**При КАЖДОМ обновлении версии (PROJECT_VERSION_PATCH в CMakeLists.txt) — ОБЯЗАТЕЛЬНО:**

1. **Актуализируй раздел 21.1 «Текущее состояние реализации»:**
   - Обнови статус всех требований (3.2.4–3.2.9 и других)
   - Отметь какие требования теперь ✅ РЕАЛИЗОВАНО (были ❌ или ⚠️)

   - Обнови таблицу сборки (версия, компилятор, библиотеки, опции)

2. **Обнови таблицу задач (раздел 21.4):**
   - Добавь новые задачи, которые были выполнены с последней версии
   - Укажи версию выполнения для каждой новой задачи
   - Обнови общий итог по часам и статус

3. **Обнови версию в документах:**
   - `README.md`: бейдж версии, таблица параметров, футер
   - `translations/ru.po`: `Project-Id-Version`
   - `TECHNICAL_SPECIFICATION.md`: раздел 21.1 «Текущая версия сборки»
   - Все другие файлы где версия упоминается явно

4. **Проверь консистентность версий:**
   - Все файлы должны содержать одинаковую версию
   - Не должно быть расхождений между README, spec, CMakeLists.txt, .po

**Формат записи в 21.1:**
```markdown
| **3.2.X Название** | ✅ РЕАЛИЗОВАНО | Краткое описание реализации | v1.0.XXX |
```

**Пример триггера:** При изменении `PROJECT_VERSION_PATCH` в CMakeLists.txt с 431 на 434 —
обнови все таблицы, статусы, списки файлов и версии в разделе 21.

## Правило: Обновление SPEC-файла при изменении версии

**При КАЖДОМ обновлении `PROJECT_VERSION_PATCH` в CMakeLists.txt — ОБЯЗАТЕЛЬНО:**

1. **Обнови `mh-compressor-manager.spec`:**
   - Строка `Version:` должна совпадать с версией из CMakeLists.txt (формат MAJOR.MINOR.PATCH)
   - Примечание: `_rpm-build.sh` автоматически подставляет версию из CMakeLists.txt через sed перед сборкой, но SPEC всё равно должен быть актуален для ручной сборки и чтения

2. **Обнови `%changelog` в SPEC-файле:**
   - Добавь новую запись с текущей датой, версией и кратким описанием изменений
   - Формат: `* <день недели> <месяц> <день> <год> <Автор> <email> - <VERSION>-1`
   - Пример: `* Вт 14 апр 2026 Павел Коваленко <dev@mediahive.ru> - 1.0.455-1`

3. **Проверь консистентность:**
   - `CMakeLists.txt`: PROJECT_VERSION_MAJOR.MINOR.PATCH
   - `SPEC`: Version: MAJOR.MINOR.PATCH
   - `README.md`: бейдж, таблица, футер
   - `translations/ru.po`: Project-Id-Version
   - Все четыре должны совпадать

4. **Проверь пути в SPEC:**
   - Все относительные пути (`../../`) корректны при сборке из `src/build/`
   - `%files` содержит все устанавливаемые файлы (binary, config, man, completion, locale, doc, license)

## Правило: Подсчёт коммитов и обновление PROJECT_VERSION_PATCH перед пушем

**Перед КАЖДЫМ пушем в ветку `main` — ОБЯЗАТЕЛЬНО:**

1. **Определи текущий PROJECT_VERSION_PATCH на GitHub (origin/main):**
   ```bash
   git fetch origin
   git show origin/main:src/CMakeLists.txt | grep PROJECT_VERSION_PATCH
   ```
   Это — **базовое число** (количество уже запушенных коммитов на GitHub).

2. **Посчитай количество локальных коммитов ahead of origin/main:**
   ```bash
   git rev-list --count origin/main..HEAD
   ```
   Это — **количество новых коммитов**, которые будут запушены.

3. **Рассчитай новую версию:**
   ```
   PROJECT_VERSION_PATCH = (коммитов на origin/main) + (локальные коммиты ahead) + 1
   ```
   **+1** — это сам коммит обновления версии, который будет запушен.
   После пуша: `TOTAL == PATCH` — совпадает.

   Пример:
   ```
   origin/main: 440 коммитов
   local ahead: 0 (коммит ещё не создан)
   PATCH = 440 + 0 + 1 = 441
   После пуша: origin/main = 441, PATCH = 441 ✓
   ```

4. **Обнови файлы перед пушем:**
   - `src/CMakeLists.txt`: `set(PROJECT_VERSION_PATCH <рассчитанное_число>)`
   - `README.md`: бейдж, таблица параметров, футер
   - `translations/ru.po`: `Project-Id-Version`
   - `docs/specification/TECHNICAL_SPECIFICATION.md`: раздел 21.1

5. **Если изменения уже содержат коммит с версией — используй `git commit --amend` или создай дополнительный коммит с обновлением версии.**

**Пример расчёта:**
```
GitHub (origin/main): PROJECT_VERSION_PATCH = 436
Локальные коммиты ahead: 2
Новая версия: 436 + 2 + 1 = 439
→ Установить PROJECT_VERSION_PATCH = 439 перед push
```

**Проверка после пуша:**
```bash
git fetch origin
TOTAL=$(git rev-list --count origin/main)
PATCH=$(git show origin/main:src/CMakeLists.txt | grep PROJECT_VERSION_PATCH | grep -oP '\d+')
# TOTAL и PATCH должны совпадать
```

**Формула перед пушем:**
```
PATCH = (коммитов на GitHub) + (локальные коммиты, включая текущий)
```
После пуша: `TOTAL = PATCH` — совпадает.

**Исключение:** Если пушится только обновление версии (без других изменений) — достаточно одного коммита, расчёт: базовое + 1.

**Важно:** PROJECT_VERSION_PATCH = количество закрытых PR (запушенных коммитов в main).
Это число должно **всегда совпадать** с реальным количеством коммитов в ветке main на GitHub.

## Правило: Обязательная проверка ссылок при изменении документации

**При ЛЮБОМ изменении файлов `.md` (добавление, редактирование, удаление разделов) — ОБЯЗАТЕЛЬНО:**

   - Нет ли дублирующих разделов с разными названиями
   - Нет ли перекрёстных ссылок которые ведут на неверные разделы

   - Битые якоря — исправь на корректные
   - Устаревшее содержание — обнови
   - Дублирующие пункты — удали или перенеси в правильный раздел
