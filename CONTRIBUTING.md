# Pull Request и коммиты — mh-compressor-manager

## Коммиты
- Сообщение коммита на **русском языке**
- Формат: `[Тип]: Краткое описание (до 72 символов)`
- Типы: `fix:`, `chore:`, `feat:`, `refactor:`, `docs:`
- Пример: `fix: устранение гонки данных при чтении m_cfg`
- Пример для текущей версии: `feat: streaming чанковое сжатие с адаптацией под L3 кэш CPU`

## Создание Pull Request

### 1. Ветка
**ВСЕГДА** публиковать изменения в отдельной ветке, не напрямую в `main`:
```bash
git checkout -b fix/описание-проблемы
git push origin fix/описание-проблемы
```

### 2. Заголовок PR
- На **русском языке**, краткий и понятный
- Формат: `[Тип изменений]: Краткое описание`
- Примеры:
  - `fix: устранение гонки данных в monitor.cpp`
  - `feat: поддержка асинхронного сжатия Brotli`
  - `chore: обновление версии до 1.0.88`

### 3. Описание PR
**Обязательно подробное описание на русском языке.** Структура:

```markdown
## Описание
Подробное описание того, что было изменено и почему.

### Проблема
Что сломалось, симптомы, как воспроизвести.

### Решение
Что сделано, какие файлы затронуты.

### Результат
- ✅ Сборка: без ошибок
- ✅ Тесты: все прошли
- ✅ Сервис: active (running)
```

### 4. Создание PR через gh api с JSON

**ВАЖНО:** При использовании `--field` GitHub API обрезает многострочное тело PR. Чтобы описание не обрезалось, **всегда отправляй данные через JSON-файл**.

**Шаг 1:** Создать файл с JSON-телом запроса:
```bash
cat > /tmp/pr_body.json << 'ENDJSON'
{
  "title": "fix: краткое описание на русском",
  "head": "имя-ветки",
  "base": "main",
  "body": "## Описание\n\nПодробное описание на русском языке...\n"
}
ENDJSON
```

**Шаг 2:** Отправить PR через `gh api` с `--input`:
```bash
gh api repos/pavelvkovalenko/mh-compressor-manager/pulls \
  --method POST \
  --input /tmp/pr_body.json
```

**Почему не --field:** `--field` передаёт данные как form-urlencoded, GitHub API обрезает длинные и многострочные значения. `--input` с JSON гарантирует что тело передаётся целиком.

### 5. Обязательная проверка созданного PR

После создания PR через `gh api` **обязательно** проверить его наличие и содержимое:

```bash
gh pr view <номер> --json title,body,headRefName,baseRefName,state
```

Это необходимо потому что:
- Из-за сбоев сети или API PR может не сохраниться
- GitHub может вернуть уже существующий PR по другому вопросу
- Описание (body) может быть обрезано или не примениться
- Нужно убедиться что ветка, заголовок и описание соответствуют ожидаемым

Если PR не создан или содержит неверные данные — повторить создание.

## Структура PR описания
```markdown
## 📋 Описание
Подробное описание того, что было изменено и почему.

## 🔧 Тип изменений
- [ ] 🐛 Исправление ошибки (bug fix)
- [ ] ✨ Новая функциональность (feature)
- [ ] ⚡ Оптимизация производительности (performance)
- [ ] 📝 Документация (documentation)
- [ ] 🔧 Рефакторинг (refactoring)
- [ ] 🧪 Добавлены тесты (tests)

## 📁 Изменённые файлы
- `src/file1.cpp` - краткое описание изменений
- `src/file2.h` - краткое описание изменений

## ✅ Тестирование
Описание проведённого тестирования.

## 📸 Скриншоты (если применимо)
Скриншоты или примеры вывода до/после изменений.
```
