## 2. Назначение и принципы работы

> **✅ Основная цель:** Автоматизация процесса сжатия статических файлов для оптимизации работы веб-сервера.

### 2.1. Мониторинг

Программа отслеживает изменения в файловой системе в режиме реального времени (мониторинг).

### 2.2. Обработка файлов

При обнаружении новых или измененных файлов с подходящими расширениями программа создает их сжатые копии в той же директории.

### 2.3. Алгоритмы сжатия

| Алгоритм | Расширение | Библиотека |
|----------|-----------|------------|
| **Gzip** | `.gz` | `zlib` |
| **Brotli** | `.br` | `libbrotli` |

### 2.4. Сохранность данных

Исходные файлы **не удаляются** и **не модифицируются**.

### 2.5. Метаданные

Сжатые копии должны наследовать от оригинала:
- **Время изменения (mtime)** — обязательно (критично для HTTP-кэширования)
- **Права доступа (permissions)** — обязательно
- **Контекст безопасности SELinux** — обязательно (если применимо)
- **Владельца и группу (UID/GID)** — обязательно

**Время создания (birth time)** сжатого файла должно соответствовать **реальному текущему времени** его создания (не наследуется от оригинала). Это полезно для аудита и отслеживания того, когда именно было выполнено сжатие.

### 2.6. Архитектура

| Компонент | Назначение |
|-----------|-----------|
| **Основной процесс** | Демон, работащий в фоне |
| **Поток-диспетчер** | Обрабатывает события файловой системы (inotify) |
| **Пул потоков** | Выполняет операции сжатия асинхронно |
| **Systemd-интеграция** | Взаимодействие через уведомления о готовности (`sd_notify`) |

## 3. Требования к функциональности

### 3.1. Мониторинг файловой системы

#### 3.1.1. Рекурсивное сканирование

Программа должна рекурсивно сканировать указанные целевые директорторию(ии).

#### 3.1.2. Механизм отслеживания

Используется механизм `inotify` (API или библиотека) для отслеживания событий:
- Создание файлов
- Модификация файлов
- Удаление файлов

#### 3.1.3. Обработка событий

| Событие | Действие |
|---------|----------|
| **Создание/Изменение** | Если файл имеет подходящее расширение, планируется задача на сжатие |
| **Удаление** | Если удаляется исходный файл, соответствующие сжатые копии (`.gz`, `.br`) также должны быть удалены |
| **Игнорирование** | Программа не должна следовать за символическими ссылками (symlinks) |
| **Фильтрация** | Не обрабатывать файлы, которые уже являются сжатыми копиями (оканчиваются на `.gz` или `.br`) |

#### 3.1.4. Debounce (Задержка)

> **📌 Требование:** Для предотвращения множественной обработки одного файла при частых изменениях (например, при копировании), сжатие выполняется только после того, как файл не изменялся в течение заданного интервала (параметр `debounce_delay`).

#### 3.1.5. Идемпотентность

Если сжатый файл уже существует и его время модификации новее или равно времени модификации исходного файла, повторное сжатие не производится (при первоначальном сканировании).

#### 3.1.6. Очистка устаревших (stale) сжатых файлов

**Проблема:** Возможна ситуация, при которой исходный файл был больше минимального порога сжатия, но затем уменьшился и стал ниже порога. При этом содержимое файла изменилось, возможно даже в то время, когда программа не работала. В результате сжатый файл содержит данные, не соответствующие текущему содержимому оригинала, хотя время модификации может совпадать (если файл был изменён externally и затем восстановлен с теми же метаданными).

**Пример сценария:**
```
Шаг 1: file.txt = 1025 байт  →  сжат в file.txt.gz (150 байт)
Шаг 2: file.txt изменён externally, стал 171 байт (< порога 256)
Шаг 3: Программа запускается, видит:
       - file.txt: 171 байт < 256 → НЕ сжимает
       - file.txt.gz: 150 байт, mtime = старый
       - Содержимое file.txt.gz НЕ соответствует file.txt! ← ПРОБЛЕМА!
```

**Требование:** При первоначальном сканировании и при обработке событий программа должна:

1. **Обнаруживать stale-файлы:** Если исходный файл существует, но его размер ниже минимального порога сжатия (`min_compress_size`), при этом сжатые копии (`.gz`, `.br`) существуют — сжатые копии считаются **недействительными** и подлежат удалению.

2. **Проверять соответствие содержимого:** Даже если mtime исходного файла совпадает с mtime сжатого файла, но размер исходного файла ниже порога — сжатый файл **должен быть удалён**, поскольку его содержимое может не соответствовать оригиналу.

3. **Алгоритм проверки при сканировании:**
```
Для каждого найденного исходного файла:
  ЕСЛИ file.size < min_compress_size:
    ЕСЛИ file.gz ИЛИ file.br существуют:
      УДАЛИТЬ file.gz И file.br
      ЛОГИРОВАТЬ: "Удалён stale-сжатый файл (оригинал ниже порога)"
    ПРОДОЛЖИТЬ (не сжимать)
  ИНАЧЕ:
    Выполнить стандартную stale-check по mtime
```

4. **Алгоритм при событии изменения:**
```
При событии модификации file.txt:
  ЕСЛИ file.size < min_compress_size:
    ЕСЛИ file.gz ИЛИ file.br существуют:
      УДАЛИТЬ file.gz И file.br
      ЛОГИРОВАТЬ: "Удалён stale-сжатый файл после модификации"
    ПРОДОЛЖИТЬ (не сжимать)
  ИНАЧЕ:
    Запланировать сжатие
```

5. **Логирование:** Удаление stale-файлов из-за того что оригинал стал ниже порога, должно логироваться на уровне `INFO`:
```
[INFO] Удалён stale-сжатый файл /srv/www/file.txt.gz: исходный файл 171 байт < порога 256 байт
```

#### 3.1.7. Обработка перемещений (move/rename) файлов и папок

**Проблема:** Механизм `inotify` генерирует пары событий `IN_MOVED_FROM` + `IN_MOVED_TO` при перемещении/переименовании файлов внутри monitored-директории и между директориями. Однако:
- Если файл перемещён **из** monitored-директории — приходит только `IN_MOVED_FROM` (или `IN_DELETE`)
- Если файл перемещён **в** monitored-директорию — приходит только `IN_MOVED_TO` (или `IN_CREATE`)
- При переименовании **внутри** monitored-директории — приходит пара `IN_MOVED_FROM` + `IN_MOVED_TO` с одинаковым cookie
- Если программа не работала в момент перемещения — stale-файлы остаются на старом месте
- При перемещении **папки** — inotify может не отследить рекурсивно все файлы внутри

**Требования к обработке перемещений:**

1. **Переименование файла внутри monitored-директории:**
   - Обнаружить пару `IN_MOVED_FROM` → `IN_MOVED_TO` (по cookie)
   - Удалить старые сжатые копии со **старого** пути
   - Создать новые сжатые копии на **новом** пути (если файл выше порога)
   - Если файл ниже порога — только удалить старые копии

2. **Перемещение файла ИЗ monitored-директории:**
   - При событии `IN_MOVED_FROM` или `IN_DELETE`: удалить сжатые копии (`.gz`, `.br`)
   - Не пытаться сжать файл (его больше нет в monitored-зоне)

3. **Перемещение файла В monitored-директорию:**
   - При событии `IN_MOVED_TO` или `IN_CREATE`: обработать как новый файл
   - Если файл выше порога — сжать
   - Если файл ниже порога — не сжимать, проверить наличие stale-копий

4. **Перемещение папки внутри monitored-директории:**
   - При перемещении папки — все файлы внутри неё должны быть переобработаны
   - Старые сжатые копии в исходной папке — удалены
   - Новые файлы на новом месте — обработаны (сжаты или пропущены по размеру)

5. **Перемещение папки ИЗ monitored-директории:**
   - Все сжатые копии внутри перемещённой папки должны быть удалены
   - Рекурсивный обход: удалить все `.gz` и `.br` в перемещённой папке

6. **Перемещение папки В monitored-директорию:**
   - Рекурсивное сканирование перемещённой папки
   - Обработка всех подходящих файлов (сжатие или пропуск по размеру)
   - Добавление новых inotify-watch на поддиректории

7. **Перемещение при неработающей программе:**
   - При запуске (initial scan) обнаружить stale-файлы на **старых** путях и удалить их
   - Обнаружить файлы на **новых** путях и обработать их
   - Проверка: stale-файл существует, но соответствующего исходного файла нет → удалить

8. **Корректность cookie для пары MOVED_FROM/MOVED_TO:**
   - События с одинаковым cookie относятся к одному перемещению
   - Если cookie не совпадает или одно событие из пары потеряно — обработать оба события независимо (как удаление + создание)

### 3.2. Алгоритмы сжатия

#### 3.2.1. Уровень сжатия

Уровень сжатия для Gzip и Brotli должен быть настраиваемым.

#### 3.2.2. Многопоточность

Поддержка многопоточности: параллельное сжатие разных файлов через пул потоков.

#### 3.2.3. Ограничение нагрузки

Количество параллельных операций сжатия не должно превышать заданное значение (параметр `threads`).

### 3.3. Расширения файлов

#### 3.3.1. Список расширений по умолчанию

```
txt, js, css, svg, json, html, htm, map
```

#### 3.3.2. Регистронезависимость

Сравнение расширений должно быть регистронезависимым (`.JS` = `.js`).

#### 3.3.3. Пользовательская конфигурация

Пользователь может изменять список расширений в конфигурации.

## 4. Требования к минимальному размеру файла для сжатия

> **📋 Основание:** Анализ индустриальных практик и технических исследований (см. раздел [15. Обоснование требований](#15-обоснование-требований-к-минимальному-размеру-файла)).

### 4.1. Пороговые значения

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| `MIN_COMPRESS_SIZE` | **256 байт** | Файлы менее 150 байт могут увеличиться после сжатия; 256 байт — практический оптимум |
| `OPTIMAL_COMPRESS_SIZE` | **1024 байт (1 КБ)** | Файлы ≥1 КБ дают значительный выигрыш от сжатия |

### 4.2. Стратегия сжатия по размерам файлов

| Размер файла | Действие | Приоритет |
|--------------|----------|-----------|
| **< 256 байт** | **Не сжимать** | Высокий |
| **256–1023 байт** | **Сжимать** | Средний |
| **≥ 1024 байт** | **Сжимать обязательно** | Высокий |

### 4.3. Требования к логированию малых файлов

При пропуске файлов размером менее 256 байт система должна:
- Логировать событие на уровне `DEBUG` (не `WARNING` и не `ERROR`)
- Указывать причину: `file size below minimum threshold (256 bytes)`
- Не создавать предупреждения для пользователя

**Пример:**
```
[DEBUG] Пропуск файла /srv/www/small.js: размер 128 байт < минимального порога (256 байт)
```

### 4.4. Требования к конфигурации

Параметры должны быть настраиваемыми через конфигурационный файл:

```ini
[compression]
min_compress_size = 256
optimal_compress_size = 1024
```

**Требования:**
- Значения по умолчанию: 256 и 1024 байт соответственно
- Минимально допустимое значение `min_compress_size`: 128 байт
- Минимально допустимое значение `optimal_compress_size`: 512 байт
- При некорректных значениях использовать умолчания с логированием предупреждения

### 4.5. Проверка размера файла

**Псевдокод:**
```cpp
bool should_compress(size_t file_size) {
    if (file_size < m_cfg.min_compress_size) {
        return false;
    }
    return true;
}
```

**Требования:**
- Проверка выполняется **до** начала сжатия
- Проверка выполняется **после** открытия файла и получения его размера
- Не пытаться сжимать файлы ниже порога

### 4.6. Статистика пропущенных файлов

Система должна вести счётчики:
- `files_skipped_small` — количество пропущенных файлов (размер < порога)
- `bytes_skipped_small` — общий объём пропущенных данных (байт)

Эти счётчики должны быть доступны в:
- Выводе статуса сервиса
- Метриках (если реализовано)
- Логах статистики при завершении работы

### 4.7. Предварительное сжатие в фоне

**Контекст:** mh-compressor-manager выполняет сжатие **предварительно** (не на лету) с использованием **максимальной степени сжатия**:
- **gzip:** уровень 9
- **brotli:** уровень 11

**Вывод:** Затраты процессора на сжатие не являются критичными, так как выполняются в фоне. Однако распаковка на клиенте остаётся фактором, поэтому порог 256 байт сохраняет актуальность.

### 4.8. Совместимость с nginx

Настройки mh-compressor-manager должны быть совместимы с рекомендациями nginx:
- `gzip_min_length 256;`
- `brotli_min_length 256;`

Это обеспечивает единообразие обработки файлов на обоих уровнях.

## 5. Требования к конфигурации

| Параметр | Значение |
|----------|----------|
| **5.1. Путь к конфигурационному файлу** | `/etc/mediahive/compressor-manager.conf` |
| **5.2. Формат** | INI |
| **5.3. Приоритет настроек** | Аргументы командной строки имеют приоритет над конфигурационным файлом |

### 5.4. Структура файла конфигурации

```ini
[general]
# Каталоги для мониторинга (разделитель ; или пробел)
target_path=/var/www/html;/var/www/site1

# Логирование отладки (true/false)
debug=false

# Количество потоков (0 = авто)
threads=0

# Список расширений (без точек, через пробел)
list=txt js css svg json html htm map

# Алгоритмы (gzip, brotli, all)
algorithms=all

# Уровень сжатия Gzip (1-9)
gzip_level=6

# Уровень сжатия Brotli (1-11)
brotli_level=4

# Задержка перед сжатием после изменения файла (сек)
debounce_delay=2

# Минимальный размер файла для сжатия (байт)
min_compress_size=256

# Оптимальный размер файла для сжатия (байт)
optimal_compress_size=1024

# Опционально: переопределение настроек для конкретной папки
[/var/www/site2]
list=html htm
algorithms=all
gzip_level=6
brotli_level=4
```

### 5.5. Отсутствие конфигурационного файла

## 15. Обоснование требований к минимальному размеру файла

### 15.1. Консенсус индустрии

| Источник | Рекомендуемый минимум | Обоснование |
|----------|----------------------|-------------|
| **Google** | 150–1000 байт | Файлы <150 байт могут увеличиться после сжатия |
| **Akamai (CDN)** | 860 байт | Файлы до 860 байт помещаются в один TCP-пакет, сжатие не даёт сетевых преимуществ |
| **GetPageSpeed** | 256 байт | Практический оптимум для веб-серверов, рекомендован для nginx |
| **Apache Tomcat** | 2 КБ | Консервативный подход для серверов приложений |
| **CubePath** | 1 КБ | Накладные расходы не окупаются для файлов <1KB |
| **OneUptime** | 1 КБ (Brotli) | Пропуск файлов <1KB для Brotli-сжатия |
| **StackExchange (сообщество)** | 150–1000 байт | На основе практического тестирования |

### 15.2. Технические накладные расходы

#### GZIP overhead

- **Заголовок:** 10 байт
- **Футер (CRC32 + размер):** 8 байт
- **Итого служебных данных:** 18 байт
- **Словари Хаффмана для малых файлов:** ~50–65 байт
- **Общий overhead:** 18–83 байта в зависимости от содержимого

#### Риск увеличения файла

- Файлы **<150 байт** после сжатия gzip могут стать **больше оригинала**
- **Причина:** служебные заголовки + словарь сжатия превышают выигрыш от алгоритма DEFLATE
- **Механизм предотвращения:** DEFLATE автоматически формирует блоки типа 0 (Uncompressed), когда сжатие не даёт выгоды

#### Сетевые соображения

- **Один TCP-пакет:** файлы до ~860 байт помещаются в один пакет (MTU ~1500 байт с учётом заголовков)
- **Вывод Akamai:** сжатие файлов, помещающихся в один пакет, не даёт заметных сетевых преимуществ

### 15.3. Практические результаты тестирования

#### Тест 1: пользователь Yura (StackExchange)
- **Метод:** Проверка порога 2 КБ
- **Результат:** Выходной размер файла практически не уменьшается, добавляются временные затраты на сжатие
- **Рекомендация:** Поднять порог до 3–4 КБ

#### Тест 2: пользователь utt73 (StackExchange)
- **Порог:** ~350 байт
- **Результат:** Сжатие AJAX-запросов в диапазоне 350–1000 байт
- **Вывод:** Прямой выигрыш в производительности минимален, но снижаются затраты на CDN
- **Итог:** В сочетании с кэшированием достигнуты показатели загрузки страницы и Time to Interactive (TTI) < 2 секунд

#### Тест 3: библиотека jQuery (GetPageSpeed)
- **Исходный размер:** 89 КБ
- **После Brotli:** 27 КБ
- **Экономия:** ~70% по сравнению с оригиналом, ~13% лучше gzip

### 15.4. Рекомендуемые пороги

| Размер файла | Стратегия | Обоснование |
|--------------|-----------|-------------|
| **<150 байт** | ❌ Не сжимать | Файлы могут увеличиться |
| **150–255 байт** | ⚠️ Пограничная зона | Выигрыш минимален (10–30 байт), распаковка добавляет задержку |
| **256–1023 байт** | ✅ Сжимать | Умеренный выигрыш, стабильное сжатие |
| **≥1024 байт (1 КБ)** | ✅✅ Обязательно | Значительный выигрыш, оптимальная зона |

### 15.5. Почему выбран порог 256 байт для mh-compressor-manager

✅ **Предварительное сжатие** (не на лету) — затраты на процессор не важны  
✅ **Максимальная степень сжатия** — brotli level 11 эффективнее даже для малых файлов  
✅ **Распаковка на клиенте** — дешёвая операция, но для файлов <256 байт время распаковки сравнимо с экономией трафика  
✅ **Сетевые пакеты** — файлы до ~860 байт помещаются в один TCP-пакет, но 256+ байт уже дают экономию 30–70%  
✅ **Совместимость с nginx** — согласуется с настройками по умолчанию (`gzip_min_length 256;`, `brotli_min_length 256;`)

## 17. Обоснование требований к временным меткам файлов

### 17.1. Исследование: какие временные метки использует nginx

#### Типы временных меток файлов в Linux

| Метка | Описание | Используется nginx? |
|-------|----------|---------------------|
| **mtime** (modification time) | Время последнего изменения **содержимого** файла | ✅ **ДА** — основная |
| **ctime** (change time) | Время изменения **метаданных** (права, владелец) | ❌ Нет |
| **atime** (access time) | Время последнего **чтения** файла | ❌ Нет |
| **birth time** (creation time) | Время **создания** файла (inode) | ❌ Нет |

#### Как nginx генерирует HTTP-заголовки для статических файлов

**Last-Modified:**
```c
// Исходный код nginx
r->headers_out.last_modified_time = of.mtime;
```
Используется исключительно **mtime** файла.

**ETag:**
```
ETag = "%xT-%xO"  →  "62b1fab3-267"
         ↑     ↑
      mtime  Content-Length
```
ETag формируется из **mtime + размер файла** (в шестнадцатеричном формате).

#### Механизм отдачи файлов nginx

```
Диск → stat() → проверка mtime → sendfile() → сеть
                  ↑
          Last-Modified + ETag
```

nginx использует системный вызов `stat()`, который считывает `mtime`, но **не** `birth time`. Директива `sendfile` передаёт файл напрямую из дискового кэша ядра в сетевой сокет — время создания не участвует.

### 17.2. Выводы исследования

**Время создания (birth time) НЕ влияет на отдачу nginx/CDN**, поскольку:
1. nginx не передаёт birth time в HTTP-ответах
2. CDN-провайдеры (Cloudflare, Akamai и др.) кэшируют файлы на основе HTTP-заголовков (`Cache-Control`, `ETag`, `Last-Modified`), а не файловых метаданных
3. Механизм `sendfile` не использует birth time

**Время изменения (mtime) КРИТИЧНО**, поскольку:
1. Определяет заголовок `Last-Modified` в HTTP-ответе
2. Участвует в генерации `ETag`
3. Влияет на кэширование в браузерах и CDN
4. Определяет валидность кэша при ревалидации (`If-Modified-Since`)

### 17.3. Требования к обработке временных меток

#### mtime — наследовать от оригинала

| Параметр | Требование | Причина |
|----------|-----------|---------|
| **mtime сжатого файла** | Должен совпадать с mtime оригинала | nginx использует mtime для `Last-Modified` и `ETag`. Если сжатый файл будет иметь другой mtime, nginx отдаст другие кэш-заголовки, что приведёт к некорректному кэшированию и сбросу кэша у пользователей |

**Пример проблемы при неправильном mtime:**
```
Оригинал style.css:  mtime = 2026-04-10 12:00:00
Сжатый style.css.br:  mtime = 2026-04-11 08:00:00  ← НЕВЕРНО!

nginx отдаёт для style.css:     Last-Modified: Sat, 10 Apr 2026 12:00:00 GMT
nginx отдаёт для style.css.br:  Last-Modified: Sun, 11 Apr 2026 08:00:00 GMT  ← РАЗНЫЕ!

Результат: браузер не может корректно кэшировать пару файлов
```

#### birth time — не наследовать, устанавливать текущее

| Параметр | Требование | Причина |
|----------|-----------|---------|
| **birth time сжатого файла** | Текущее реальное время создания | Полезно для аудита: администратор может определить, когда именно было выполнено сжатие. Не влияет на HTTP-ответы nginx |

**Пример для аудита:**
```
$ stat style.css style.css.br

  File: style.css
    Birth: 2026-04-01 10:00:00  ← оригинал создан ранее
    Modify: 2026-04-10 12:00:00

  File: style.css.br
    Birth: 2026-04-11 08:30:15  ← сжатие выполнено сегодня
    Modify: 2026-04-10 12:00:00  ← mtime совпадает с оригиналом
```

## 18. Требования к тестированию

### 18.1. Общие требования

Тестирование проводится в формате **автоматизированного цикла**: тестирование → поиск ошибок → исправление → повторное тестирование до полного устранения всех дефектов.

**Принципы тестирования:**
- Все тесты выполняются на целевой платформе (`www` — удалённый сервер Fedora)
- Найденные ошибки исправляются немедленно, после чего тесты запускаются заново
- Цикл повторяется до достижения **нулевого количества ошибок**
- Результаты всех тестов документируются

### 18.2. Нагрузочное тестирование

#### 18.2.1. Требования к тестовым данным

| Параметр | Значение |
|----------|----------|
| **Объём данных** | Не менее **1 ГБ** |
| **Количество файлов** | Не менее **300 файлов** разных типов |
| **Предварительно сжатые файлы** | Не более **50%** от общего количества |
| **Типы файлов** | HTML, CSS, JS, JSON, XML, TXT, SVG |
| **Размеры файлов** | От 256 байт до 100 МБ (разнообразие) |

**Цель:** Проверка производительности и стабильности при обработке большого объёма данных.

#### 18.2.2. Требования к сценарию нагрузочного теста

1. **Initial Scan (первоначальное сканирование):**
   - Запуск программы с пустым состоянием
   - Измерение времени сканирования всех файлов
   - Измерение времени сжатия всех подходящих файлов
   - Мониторинг потребления памяти (ОЗУ) в процессе
   - Фиксация пиковой нагрузки на CPU

2. **Тест механизма stale-check (повторное сжатие):**
   - Модификация 20-30% исходных файлов после начального сжатия
   - Перезапуск программы или ожидание обнаружения изменений
   - Проверка что изменённые файлы были пережаты
   - Проверка что не изменённые файлы **НЕ** были пережаты (идемпотентность)
   - Измерение времени повторной обработки

3. **Тест удаления stale-файлов при уменьшении исходника:**
   - Создать файлы >256 байт, сжать их
   - Уменьшить размер 10-15 файлов до <256 байт (ниже порога) с изменением содержимого
   - Перезапустить программу
   - **Проверить:** Все `.gz` и `.br` копии уменьшённых файлов **удалены**
   - **Проверить:** В логах есть записи `INFO` об удалении stale-файлов
   - **Проверить:** Файлы, оставшиеся выше порога, НЕ затронуты

4. **Тест удаления файлов:**
   - Удаление 10-15% исходных файлов
   - Проверка что соответствующие `.gz` и `.br` копии были удалены
   - Проверка целостности оставшихся файлов

5. **Тест добавления новых файлов:**
   - Добавление 50+ новых файлов во время работы демона
   - Проверка что все новые файлы были обнаружены и сжаты
   - Измерение задержки между добавлением и сжатием (debounce)

6. **Тест перемещений и переименований файлов:**
   - Переименование 20 файлов внутри monitored-директории
   - Перемещение 10 файлов из monitored-директории наружу
   - Перемещение 10 файлов извне в monitored-директорию
   - Перемещение папки с 30 файлами внутри monitored-директории
   - Перемещение папки с 20 файлами из monitored-директории наружу
   - Перемещение папки с 20 файлами извне в monitored-директорию
   - **Проверить:** Все stale-копии на старых путях удалены
   - **Проверить:** Все файлы на новых путях сжаты (если выше порога)
   - **Проверить:** Нет дубликатов сжатых файлов
   - **Проверить:** Нет orphan-файлов (`.gz`/`.br` без оригинала)

#### 18.2.3. Метрики нагрузочного теста

| Метрика | Единица | Фиксируется |
|---------|---------|-------------|
| Время initial scan | секунды | ✅ |
| Время сжатия всех файлов | секунды/минуты | ✅ |
| Пиковое потребление ОЗУ | МБ | ✅ |
| Среднее потребление ОЗУ | МБ | ✅ |
| CPU usage (пик) | % | ✅ |
| Количество сжатых файлов | шт | ✅ |
| Количество пропущенных файлов (< 256 байт) | шт | ✅ |
| Коэффициент сжатия (итого) | % | ✅ |

### 18.3. Тестирование адаптивного управления памятью

#### 18.3.1. Требования к поведению пулов памяти

**Во время всплеска нагрузки:**
- Если свободная память на устройстве позволяет — система **должна увеличить** лимиты пулов памяти для достижения максимальной производительности
- Увеличение лимитов происходит **автоматически** при обнаружении свободной памяти
- Программа должна логировать факт увеличения лимитов (уровень `INFO`)

**После падения нагрузки:**
- Размер буферного пула **обязан уменьшиться** до базового значения
- Уменьшение происходит **автоматически** после завершения обработки очереди
- Программа должна логировать факт уменьшения лимитов (уровень `INFO`)

**Оптимизация при большой очереди:**
- Если в очереди на сжатие много файлов — **НЕ** освобождать пул после каждого сжатого файла
- Причина: затраты на выделение/освобождение памяти снижают общую производительность
- Сжатие пулов должно происходить **периодически** или при превышении лимита:
  - **Вариант A:** по таймеру (например, раз в 5 минут)
  - **Вариант B:** при превышении максимального размера пула
  - **Вариант C:** при отсутствии задач в очереди в течение N секунд

#### 18.3.2. Сравнительное тестирование стратегий управления памятью

**Требование:** Провести сравнение производительности следующих стратегий:

| Стратегия | Описание | Измеряемые метрики |
|-----------|----------|-------------------|
| **A. Немедленное освобождение** | Освобождение буфера после каждого сжатого файла | Общее время, CPU, память |
| **B. Периодическое освобождение (таймер)** | Освобождение раз в N минут (рекомендуется: 5 мин) | Общее время, CPU, память |
| **C. Освобождение при превышении лимита** | Освобождение только при достижении максимального размера пула | Общее время, CPU, память |
| **D. Гибридная (таймер + лимит)** | Освобождение при превышении лимита ИЛИ раз в N минут | Общее время, CPU, память |

**Результат сравнения** должен быть задокументирован в описании проекта с указанием:
- Рекомендуемой стратегии
- Обоснования выбора
- Измеренных значений для каждой стратегии
- Сценариев, при которых каждая стратегия наиболее эффективна

> **⚠️ Важно:** Результаты сравнения необходимо сохранить в документации проекта, чтобы в будущем случайно не удалить функционал, который необходим для повышения эффективности программы.

#### 18.3.3. Метрики тестирования памяти

| Метрика | Единица | Фиксируется |
|---------|---------|-------------|
| Базовый размер пула | МБ | ✅ |
| Пиковый размер пула | МБ | ✅ |
| Время увеличения лимитов | секунды от старта | ✅ |
| Время уменьшения пула | секунды после завершения | ✅ |

| **9** | Запустить программу | stale удалены, новые файлы сжаты, целостность сохранена |
| **10** | Проверить отсутствие orphan-файлов | Нет `.gz`/`.br` без соответствующего оригинала |

### 18.7. Тестирование на утечки и баги

#### 18.7.1. Поиск утечек памяти

| Инструмент | Цель | Запуск |
|-----------|------|--------|
| **Valgrind (memcheck)** | Утечки памяти, некорректные обращения | `valgrind --leak-check=full ./mh-compressor-manager --dry-run` |
| **AddressSanitizer** | Ошибки работы с памятью (сборка с `-fsanitize=address`) | При сборке |
| **LeakSanitizer** | Утечки памяти (сборка с `-fsanitize=leak`) | При сборке |

#### 18.7.2. Поиск гонок данных (data races)

| Инструмент | Цель | Запуск |
|-----------|------|--------|
| **ThreadSanitizer** | Гонки данных в многопоточном коде (сборка с `-fsanitize=thread`) | При сборке |
| **Ручная проверка** | Корректность `std::shared_lock` / `std::unique_lock` | Code review |
